Before moving on to the next linear data structures, let's have a look at the complexity of the main operations associated to a linked list: insert, delete, and search. In the previous lectures, we have studied in detail the operations insert and delete. We did not study the operations search explicitly. But you can easily see how to implement it, because the operation delete must first search for the number to be deleted. So the operation search is embedded in the operation delete. Let's start with insert. The complexity of insert depends on whether you insert the new node at the start, the end, or at any arbitrary position of the list. If you insert at the beginning of the list, you have to create the new node, make the new node point to the first element of the list. This is done by copying the content of head to the address content of the new node, and make head point to the new note. Everyone of this instructions takes a constant time to be executed. So inserting a new node at the start of the list has a time complexity of Theta 1 in any case. Now, if you insert a node at the end of the list, you have to create the new node, traverse the list until you get to the last node and make the last node point to the new node. The creation of the new node, I'm making it point to the last node of the list, takes a constant time. But traversing the whole list takes n operations, where n is the number of nodes of the list. So inserting a node at the end of the list has a time complexity of Theta n in any case. The third and final case is when you need to insert a new node at an arbitrary position of the list. Usually this happens when you need to build a sorted list. That is, as you insert new nodes into the list, they must go to a specific position to keep the list sorted from smallest to largest, or the other way round. In this case, you must create the new node, traverse the list until you find the correct position for the new node, insert the new node by making the new node point to the node that should appear after it in the list, and the node in the previous position must now point to the new node. Creating the node and inserting it into the list takes a constant time. But traversing the list might take just one operation if the node must be inserted in the first position or n operations if the new node must be inserted in the last position. Therefore, the time complexity of inserting a node in a sorted list is Theta 1 in the best case, and Theta n in the worst case. Now, what is the time complexity of delete? Well, it depends on where the number you want to delete is. If the number you want to delete is in the first node of the list, then you only have to make head point to the second element of the list. As no other node is pointing to the first node of the list, depending on the programming language used, the first node will be automatically deleted. Notice that we're assuming automatic deletion here. Changing the memory address toward the head points to it takes a constant time, and therefore deleting the first element of the list has a time complexity of Theta 1. If the number you want to delete is in the last element of the list, you must traverse the list until you find the number in the last node and change the node previous to the last point to null. We are assuming that because it's not pointed at by any other node, the last node is automatically deleted from memory. All instructions take a constant time, but the ones inside the while loop are repeated n times. So deleting the last node of a list has a time complexity of Theta n. So depending on the position of the list where the number to be deleted is stored, delete might take a time complexity of Theta 1 in the best case, which happens when the number is in the very first position, or it might take a time complexity of Theta n in the worst case, which happens when the number is in the last position or it's not in the list at all. Finally let's analyze the search operation. First, notice that in a linked list, you are forced to do a linear search because the structure of the list does not allow you to jump from one position to another as is the case with binary search. Because the linked list also does not allow you to access any node using indices, you cannot use direct addressing or hashing. Given this, searching in a linked list has the same time complexity as using a linear search in an array. In the best case, the time complexity is Theta 1, which is when the element you are searching for is in the first node of the list. In the worst case, the time complexity is Theta n, which is when the element you are searching for is in the last position of the list, or it's not in the list at all. Now that you have seen the main operations associated with a linked list under time complexities, let's briefly look at two other types of linked list you can build. Let's start with the doubly linked list. So far, you can traverse a single linked list from head to the end. So when you are pointing to a specific node, you can only move to the next one not the previous one. Sometimes it's useful to be able to move in both directions in the list. In this case, we build a doubly linked list where every node has three fields, data, address of the next node and address of the previous node. A second type of linked list is the circular list. In a circular list, the last node points to the first node making a circle. A good analogy to a circular list is a playlist. After playing the last song, your music player jumps to the first song again. That is exactly how a circular list works. Of course, the linear shape of a list is lost in a circular linked list. With this brief summary of the time complexity of the main operations associated with linked list, and our brief look at the other two types of linked list, we finish our theoretical study of linked list.

We have already studied how to implement a queue using an array. We saw how using an array to implement a queue has the drawback of either having unused allocated memory if the queue has a number of elements lower than the array size, or having to reject elements from joining the queue when the queue grows bigger than the array size. We also had to use circular traverse of the array when implementing a queue. Now we will see how these drawbacks disappear when using a linked list to implement a queue. First, notice that we are going to use a linked list with two pointers, front and tail. Because in a queue elements are inserted on one end, the tail, and removed from the other end, the front, if we use only one pointer, one of the operations will need to traverse the list. Traversing the list has a time complexity of Theta n. If we want to have operations with a time complexity of Theta one, we need to use two pointers. This is an example of when you get faster processing at the expense of extra memory. Let's start with an empty queue. In this case, both pointers, front and tail, point to null. Now we are going to enqueue the first element in the list. In this case, we begin by creating the new node and then we insert it at the tail of the list by making tail point to the new node. But, as in the case of the array implementation, both pointers must be updated because it's the first element to be inserted. So we also update front to point to the first node. Remember, this is only done when an element is inserted into an empty queue. This is the pseudocode for this case. We create the new node, and if the queue is empty, we make both pointers point to the new node. Now let's perform another enqueue operation to find out what steps we must carry out. We must create a new node again and we must insert it at the end of the queue. To do so, the last node that joined the queue must point to the new node and also the tail, and with that, we have finished the pseudocode description of the operation enqueue. Notice that we do not have unused memory, neither are we going to run out of the spaces as the queue grows. Finally, we do not need to take extra care in doing a circular traverse as we needed when using an array. You can simulate the insertion of new elements to the queue to check that this pseudocode is correct and to familiarize yourself with it. Now let's look at the dequeue operation. First of all, if the queue is empty, there is nothing to the queue so we display a message and return. When we dequeue, we must remove the element at the front of the queue. So the queue is equivalent to deleting an element from the start of the list. So we simply make the pointer, front, ignore the element to be deleted and point to the second element of the list. Because it's not pointed at by any other node, the first element of the list is then automatically deleted from memory. Let's do another dequeue to check the case when there is only one element in the queue. By executing the corresponding instruction, the pointer, front, will now point to null. However, tail is still pointing to the element. This is incorrect as the element should disappear from memory. So when there is only one element, that is, when front and tail point to the same node, both pointers must be updated. Here the pseudocode has been corrected to consider the situation, and because it's not being pointed at by any node or pointer, the element is deleted from memory. The operation peak simply checks if the queue is empty. If not, the data of the element at the front of the queue is returned. Finally, the operation is empty is a straightforward. If the pointers front and tail point to null, the queue is empty and the Boolean value too is returned. Otherwise, the value false is returned. As with the array implementation of a queue, if you have a quick look at the pseudocode of the different operations, you can see that all of them take a constant time. Therefore, all queue operations have a time complexity of Theta one.

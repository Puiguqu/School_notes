Bubble sort, which compares pairs of elements and swaps them if they are in the wrong order. Insertion sort, which finds the correct position in which to insert the next unsorted number in the array. Selection sort, which selects the minimum value in the unsorted part of the array and then stores it at the beginning of the unsorted part. Quicksort, which recursively selects a pivot and stores it in its final correct position with lower numbers to its left and higher numbers to its right. And Merge sort, which divides the array in halves until individual elements are left which are them merged back in sorted order. When analyzing the worst case time complexity of this algorithms, we found out that the best-performing one was merge sort with a time complexity of theta NlogN, where n is the number of elements to be sorted. And then you were told that this was the best you could get with a comparison sort in the worst case. There is no sorting algorithm based on comparisons that can have a growth better than NlogN in the worst case. in this lecture, you will get an idea of why this is so. Assume you have an array of N unsorted numbers, you might notice that there are N factorial different ways of arranging the numbers in the array. For example, for an array of three elements, there are 3 factorial, which is equal to 6, different ways of arranging the elements in the array. There are two cases where number 4 occupies the first position and the other two numbers go to the two remaining different positions. There are two cases where number 7 occupies the first position and the other two numbers go to the two remaining different positions. And there are two cases were number 1 occupies the first position and the other two numbers go to the two remaining different positions. Among all the different arrangements, only one is the correctly sorted one. In this case, is the array with the numbers 1, 4 and 7 in ascending order. The task of a sorting algorithm can be seen as finding the correct arrangement among all the different arrangements. So the key question is, what is the maximum number of comparisons a sorting algorithm must do to find the correct arrangement of numbers? To answer this question, let's visualize a small example with our three element array. The first comparison we can do here is between the first two elements of the array. Is the element in position 0 less than or equal to the element in position 1? If the answer is yes, then the two first elements are already sorted, assuming an ascending order. If the answer is no, then we need to put them in the correct order. The element originally in position 1 must go first and the element originally in position 0 must go second. Let's now move through the yes branch. In this branch, we already know that A[0] is less than or equal to A[1]. So the next question is, is the element in position 1 less than or equal to the element in position 2? If the answer is yes, we have found one of the six possible arrangements. If the answer is no, then A[2] could go either in the first or second position. Therefore, we need to make another comparison. Is A[0] less than or equal to A[2]? Depending on the answer to this question, we find two other possible arrangements. Now, let's go back to the first no branch. We now need to compare A[2] with A[0]. Is A[0] less than or equal to A[2]? If the answer is yes, we have found another possible arrangement for the numbers. If the answer is no, there are two possibilities. A[2] should be in the first position before A[1] and A[0] or in the second position between A[1] and A[0]. Therefore, we need to make another comparison. Is A[1] less than or equal to A[2]? Depending on the answer to this question, we find the two remaining arrangements for the numbers. This A structure is known as a decision tree, decision because at every step we have to make a decision in terms of yes or no, tree because different alternatives branch out from every comparison. How do we use this decision tree? Let's look at an example. If the original array contained the numbers 4, 7, and 1, the algorithm will need to make these comparisons to find the correct solution. In this case, there are three comparisons. What can be deduced from this example? First, there are N factorial possible arrangements. Each arrangement is at the very end of every possible path. In a decision tree, a terminal node is called a leaf. So there are N factorial leaves. Second, the maximum number of comparisons is equal to the length of the longest path in the decision tree. Let's call this length, L. Third, if the decision tree was complete, that is, all paths are of the same length, there will be 2 to the L leaves, this is of course a fictitious situation just to give us an upper bound to the number of leaves. Knowing that the actual number of leaves which is N factorial cannot be higher than the maximum possible number of leaves 2 to the L, we can write the following expression. That is, N factorial is less than or equal to 2 to the L. Remember that L is the length of the longest path in the decision tree, which in turn is equal to the maximum number of comparisons made by a sorting algorithm. Applying logarithm to both sides of this expression, we obtain this expression for L. That is, L is always greater than or equal to log of N factorial divided by log of 2. Using asymptotic notation, we can then say that L is Omega of log N factorial, using the Stirling approximation, N factorial can be approximated by N to the N, and this leads us to the final result. No comparison sort algorithm can be better than NlogN in the worst case. In this topic, we will study three sorting algorithms that by not using comparisons in their core processing get linear running times in the worst case.

So far all sorting algorithms treat the numbers to sort as whole. For example, if you are going to sort the numbers 24 and 13 using a comparison sort you simply compare the numbers and put them in the right order. Radix sort instead divides the numbers into their digits and sorts the numbers starting with the least significant, then the second significant digit and so on. In this case, the second significant bits are already sorted so the execution of the algorithm ends. Now that we have seen the different approach taken by Radix sort let's look at an example with a higher quantity of numbers to sort. To help you visualize the working of Radix sort I will list the numbers to sort vertically. In that way, you can easily see which digits we'll be working with. Let's start with the least significant digit. Now before sorting, you must know one important thing. The algorithm you use to sort the numbers according to the corresponding digit must be a stable sort. That is if two numbers are repeated as is the case here with numbers 7 and 8 the sorting must preserve the order in which these numbers appear in the original list. That means that when producing the new list of numbers sorted according to the least significant digit number 157 must appear before number 457. Analogously number 218 must appear before number 108. Luckily for us several sorting algorithms are stable and even better the algorithm with the best time complexity we have seen so far counting sort is stable. Okay, let's go back to sorting. We will make as many passes as there are digits in the numbers to sort. In this case, the numbers have three digits so Radix sort will make three passes. In the first pass, the numbers will be sorted according to the least significant digit. In the second pass, the list of numbers generated in the first pass will be sorted according to the second least significant digit. And in the third and final pass, the list of numbers generated in the second pass will be sorted according to the most significant digit. So after the first pass the array has the numbers in a different order. If you look at the least significant digit you can observe two things. If you use counting sort to sort the digits the time complexity of Radix sort is given by d times the complexity of counting sort which is equal to (N + k). Where d is the number of digits, N is the length of the array to be sorted, and k is the maximum value of any digit. As d is a constant number and it's only recommended to use counting sort when k is in the order of N then we can make Radix sort run a linear time with N. Another good aspect of Radix sort is that unlike counting sort you can use it not only with integer numbers but also with decimal numbers. Run an example with decimal numbers and see how it works.

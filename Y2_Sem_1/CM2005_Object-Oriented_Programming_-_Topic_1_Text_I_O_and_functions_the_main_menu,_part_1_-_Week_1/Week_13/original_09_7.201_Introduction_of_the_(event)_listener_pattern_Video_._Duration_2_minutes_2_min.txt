So I'm just going to lay out the general problem first of all and then we'll just talk a little bit about how the pattern works. So what we have is some user interface which has a button on it. The problem is, when the user comes along and clicks on the button, what are we going to do? So we need somehow to have a bit of the program which is responding to the button being clicked. The answer is events. So every time the button gets clicked, what it does, it basically triggers an event. So that's great. So we've got the idea that you click on a button and event is generated. Then the next thing is what do we do with the event? So how do we deal with the event? The answer is something called an event listener. An event listener is a common pattern in many different programming languages and API. So an event listener is something which obviously listens to events. So we have an event listener and the events get sent there. But before they get sent, it would have to register with the button to say, hey, I'm an event listener, I want to receive your events. So we just put another thing in there which is register. So what do we got? We've got a user clicking on buttons and then we've got an event which gets generated when the user clicks on the button and then we've got an event listener, which has registered itself with the button to receive events from the button. So that's really it in a nutshell. That's the event listener pattern. So we've just had a quick look at the general idea of an event listener pattern and we saw there is a certain way that data flows around triggered by the user and this idea of registering and the idea of events being generated and being received.

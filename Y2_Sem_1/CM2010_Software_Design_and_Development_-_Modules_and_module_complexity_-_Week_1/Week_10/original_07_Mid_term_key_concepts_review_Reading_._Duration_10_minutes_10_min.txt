The following resource provides a review of the key concepts covered in the first 10 weeks. You will need an understanding of these terms to complete your mid-term coursework. Topic 1: Modules and Module Complexity module: A separate component of a program that performs a specific function, promoting code organisation and reusability. modular design: The practice of dividing a system into distinct, interchangeable modules with minimal dependencies. complexity: In programming, refers to how complicated a module’s structure and interactions are, often impacting maintainability and readability. cohesion: The degree to which elements of a module work together as a single, focused unit. High cohesion indicates a well-defined purpose for the module. coupling: The level of dependency between modules. Low coupling is generally desirable as it reduces interdependencies. Types of cohesion: functional cohesion: The module performs a single, well-defined task. sequential cohesion: Actions within the module are dependent on one another in a sequence. communicational cohesion: Elements within the module work on the same data set. temporal cohesion: Tasks that occur together at the same time. procedural cohesion: Tasks are related by the control flow sequence. logical cohesion: Tasks are grouped by similarity in function but not dependency. Types of coupling: content coupling: When one module relies on the internal workings of another. common coupling: When modules share global data. control coupling: When one module controls the behaviour of another by passing information on what to do. data coupling: When modules share data through parameters but do not control each other. stamp coupling: When modules share composite data structures, not individual data elements. Topic 2: Test-driven development test-driven development (TDD): A development method where tests are written before the code, ensuring that code is driven by requirements. unit testing: Testing individual components or functions of code to verify that each works as expected. red-green-refactor: The TDD cycle: write a test and make it fail (Red), write code to pass the test (Green), and then improve (Refactor) the code. mocking: Creating “mock” objects or functions to simulate dependencies in unit tests, allowing isolated testing. test isolation: Ensuring that tests do not depend on each other or external systems, to produce consistent results. API testing: Testing Application Programming Interfaces to ensure they return expected results, often involving endpoint validation and error handling. Topic 3: Robust and secure programming (up to and including Week 10) robust programming: Writing software that can handle unexpected inputs or errors without crashing. secure programming: Coding practices that prevent security vulnerabilities like data leaks, code injection, and unauthorised access. assertion: A statement in code that verifies a condition is true, typically used for debugging or enforcing preconditions. parameter checking: Validating input parameters to ensure they meet expected conditions before processing. defensive coding: Coding strategy that anticipates and prevents errors by validating inputs, handling exceptions, and using assertions.

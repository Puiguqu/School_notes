The following resource provides a review of the key concepts you will need to know to successfully pass the mid-term exam. Topic 1: Modules and module complexity module: A separate component of a program that performs a specific function, promoting code organisation and reusability. modular design: The practice of dividing a system into distinct, interchangeable modules with minimal dependencies. complexity: In programming, refers to how complicated a module’s structure and interactions are, often impacting maintainability and readability. cohesion: The degree to which elements of a module work together as a single, focused unit. High cohesion indicates a well-defined purpose for the module. Types of cohesion: coupling: The level of dependency between modules. Low coupling is generally desirable as it reduces interdependencies. functional cohesion: The module performs a single, well-defined task. sequential cohesion: Actions within the module are dependent on one another in a sequence. communicational cohesion: Elements within the module work on the same data set. temporal cohesion: Tasks that occur together at the same time. procedural cohesion: Tasks are related by the control flow sequence. logical cohesion: Tasks are grouped by similarity in function but not dependency. Types of coupling: content coupling: When one module relies on the internal workings of another. common coupling: When modules share global data. control coupling: When one module controls the behaviour of another by passing information on what to do. data coupling: When modules share data through parameters but do not control each other. stamp coupling: When modules share composite data structures, not individual data elements. Topic 2: Test-driven development test-driven development (TDD): A development method where tests are written before the code, ensuring that code is driven by requirements. unit testing: Testing individual components or functions of code to verify that each works as expected. red-green-refactor: The TDD cycle: write a test and make it fail (Red), write code to pass the test (Green), and then improve (Refactor) the code. mocking: Creating 'mock' objects or functions to simulate dependencies in unit tests, allowing isolated testing. test isolation: Ensuring that tests do not depend on each other or external systems, to produce consistent results. API testing: Testing application programming interfaces to ensure they return expected results, often involving endpoint validation and error handling. Topic 3: Robust and secure programming robust programming: Writing software that can handle unexpected inputs or errors without crashing. secure programming: Coding practices that prevent security vulnerabilities like data leaks, code injection, and unauthorised access. assertion: A statement in code that verifies a condition is true, typically used for debugging or enforcing preconditions. parameter checking: Validating input parameters to ensure they meet expected conditions before processing. exception handling: Mechanism for handling runtime errors in code, using structures like try-catch blocks to catch and respond to errors. defensive coding: Coding strategy that anticipates and prevents errors by validating inputs, handling exceptions, and using assertions. debugger: A tool that helps programmers find and fix errors by allowing them to inspect and control program execution. conditional breakpoint: A debugger feature that pauses execution only when a specific condition is met, aiding in targeted troubleshooting. Topic 4: User testing user testing: Evaluating a program’s functionality, usability, and performance from an end-user’s perspective. user story: A description of features from the end-user’s perspective, focusing on what they should be able to do. black box testing: Testing without knowledge of the internal code, based only on input and expected output. white box testing: Testing with access to the internal workings of the code, focusing on code paths, conditions, and structures. usability testing: Testing the ease with which users can learn and use a program, often focusing on the user experience (UX). accessibility testing: Ensuring that a program is usable by people with disabilities, including compatibility with assistive technologies. Topic 5: Version control version control: System that tracks changes to code over time, allowing for collaborative work and rollback to previous versions. repository: A storage location for version-controlled code and related assets, often hosted on platforms like GitHub. clone: Creating a local copy of a remote repository. commit: Recording a set of changes to the codebase in the repository’s history. branch: A separate line of development within a repository, enabling experimentation or feature work without affecting the main codebase. merge: Combining changes from different branches into a single branch. conflict: When changes in different branches clash, requiring manual resolution before merging. gitignore: A file specifying which files or directories should not be committed to the repository, such as temporary files or sensitive information. bug reporting: Documenting issues in code, often including details about the bug, expected vs. actual behaviour, and steps to reproduce.

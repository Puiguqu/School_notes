I've got my diagram sheet here. Let me just jump out of the way and draw a diagram for you. Content coupling is when you have one module like this and then another module which is completely contained inside it. We could say that this is a very niche module coupling. It's pretty simple. The idea is that nothing outside of module one can see module two. It feels like it's quite a nice, contained, and controlled way of accessing module two and that only module 1 can see it. But it might be that module two is completely accessible to module one. It can do anything, and maybe module two has uncontrolled access to module one. It might be a little bit problematic if you're not careful how you control that. But it is a regular type of module coupling and it is used, for example, in event listeners in say JavaScript, where we pass in a complete function definition as an argument to an event listener and effectively we're defining our module on the fly and parsing it in so that event listener can only be seen from inside the event listener code. Effectively there, module one is the thing that's managing the events, and module two would be the little function that we pass in that it's going to call when an event happens. We use this pattern very commonly in many languages. You've got Lambda functions in C++ if you've ever seen those. You've got event listener classes in Java, it's another example. Many languages used this method. But one problem with it is that, if we're here, we can't see module two and that might be desirable. But what might end up happening sometimes is that you think, well, I need to be able to access module two. You end up building bridges in so that things on the outside can reach in and access module two. If you're doing that, then your engineering model is broken down and you need to probably refactor it and think more about where you need module two to be.

# What is module complexity?

So, let's dive into what module complexity is.

**What is module complexity?**

Module complexity refers to the degree of internal structure or organization within a module. In other words, it measures how well-organized and cohesive a module is.

Think about it like a team project: you might have individual team members working on separate tasks, but if they all work together seamlessly, that's a good sign of cohesion. On the other hand, if there are a lot of overlapping tasks or conflicting requirements, that's a bad sign of complexity.

In software development, module complexity is often measured by metrics such as:

1. **Cohesion**: How well-defined and focused each module is.
2. **Coupling**: How closely connected two modules are.
3. **Tangle**: The degree of interaction between modules.

So, what's the relationship between these concepts? Well, cohesion refers to how much a module "owns" its internal state, whereas coupling refers to how much a module depends on other modules. Tangle is related to coupling as well - when two modules are highly coupled (i.e., interact extensively), they can create complex tangles.

**Why is module complexity important?**

Module complexity matters because it affects the maintainability, scalability, and overall quality of your software system.

When a module is too simple or lacks cohesion, it might not be able to handle changes on its own, making it harder to update or modify. Similarly, if two modules are highly coupled, they can create complex interactions that make debugging more challenging.

On the other hand, when a module has good cohesion and minimal coupling, it becomes easier to maintain and extend. This is because each module is self-contained, reducing the risk of introducing new issues elsewhere in the system.

So, how do you measure module complexity? We'll explore some techniques for assessing module complexity throughout this course!

5. There are several types of cohesion:
	* Coupling: The level of dependency between modules (low coupling is desirable).
	* Functional cohesion: The module performs a single, well-defined task.
	* Sequential cohesion: Actions within the module depend on one another in sequence.
	* Communicational cohesion: Elements within the module work on the same data set.
	* Temporal cohesion: Tasks occur together at the same time.
	* Procedural cohesion: Tasks are related by control flow sequence.
	* Logical cohesion: Tasks are grouped by similarity in function but not dependency.

6. Test-driven development (TDD) is a development method where tests are written before code, ensuring that code is driven by requirements.
7. Unit testing verifies that individual components or functions of code work as expected.
8. The TDD cycle involves:
	* Writing a test and making it fail (Red).
	* Writing code to pass the test (Green).
	* Improving the code through refactoring.

9. Mocking is used in unit tests to simulate dependencies, allowing for isolated testing.
10. Test isolation ensures that tests do not depend on each other or external systems, producing consistent results.
11. API testing verifies that application programming interfaces return expected results, often involving endpoint validation and error handling.

12. Robust programming involves writing software that can handle unexpected inputs or errors without crashing.
13. Secure programming includes coding practices that prevent security vulnerabilities like data leaks, code injection, and unauthorized access.
14. Assertion is a statement in code that verifies a condition is true, used for debugging or enforcing preconditions.
15. Parameter checking validates input parameters to ensure they meet expected conditions before processing.

16. Exception handling involves mechanisms for handling runtime errors in code, using structures like try-catch blocks to catch and respond to errors.
17. Defensive coding anticipates and prevents errors by validating inputs, handling exceptions, and using assertions.

18. Debugging is a tool that helps programmers find and fix errors by allowing them to inspect and control program execution.
19. Conditional breakpoint is a debugger feature that pauses execution only when a specific condition is met, aiding in targeted troubleshooting.

20. User testing evaluates a program's functionality, usability, and performance from an end-user's perspective.
21. User story is a description of features from the end-user's perspective, focusing on what they should be able to do.
22. Black box testing involves testing without knowledge of the internal code, based only on input and expected output.

23. White box testing involves testing with access to the internal workings of the code, focusing on code paths, conditions, and structures.
24. Usability testing tests the ease with which users can learn and use a program, often focusing on user experience (UX).

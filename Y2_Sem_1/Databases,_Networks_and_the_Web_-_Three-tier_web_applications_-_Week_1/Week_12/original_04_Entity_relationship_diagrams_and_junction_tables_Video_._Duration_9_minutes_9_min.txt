You can refer to your reading list to learn more. Now, let me introduce you to entity relationship or ER diagrams. Drawing an ER diagram can help you to do database design. An ER diagram is a graphical representation displaying entities in a database and the relations between them. On the screen now you can see a simplified version of an ER diagram for our two table book publisher database. You can see two entities: book and publisher and a line showing the link between the two. A book is published by the publisher, and the publisher publishes a book. There are symbols at each end of the line in an ER diagram are what display their relationship cardinalities. There are four types of relationship cardinalities in an entity relationship diagram: optional-one where the line is interrupted by a circle and a vertical line. Optional-many where the line is again interrupted by a circle followed by a backward facing arrowhead. Mandatory-one, where the line is crossed by two vertical lines and finally, mandatory-many where the line is crossed by one vertical line followed by the backwards-facing arrowhead again. The important thing to note here is that optional is a may and mandatory is a must. Make sure that you read page 93 of the modern database management text book to learn in depth about the different types of cardinalities and their differences. I'll explain just one as an example. In our example, each book ID is published by mandatory-one publisher. That means each instance of book entity relates to one and only one instance of publisher entity, or in other words, each row in book table relates to one and only one role in publisher table. Note that a book title published by another publisher has a different ID number. At the other end of the line, we can see that each publisher, publishes optional-many books. Optional-many means it may publish zero to many books. This means that each instance of publisher entity can relate to zero to many instances of publisher entity, or in other words, each row in publisher table can relate to zero to many rows in book table. Here is another example of an ER diagram, this time for a restaurant. There is a food table for all the menu items and an order table. Each food can be in zero or many orders. This is optional-many cardinality. Another hand, each order contains at least one or many foods. This is mandatory many cardinality. So we can say there is a many-to-many association between their food and order tables. Unfortunately, there's a problem with many-to-many associations. The problem is that when deciding on the number of columns in a table, it is not clear how many columns should be allocated to foreign keys. In the food and order tables example, how many food IDs in the order table should be selected as foreign keys to the ID column in the food table? Shall we set a maximum number of food items for each order, say three food items per order, and allocate three columns for foreign keys? What if the customer only orders one food? Will all the remaining columns be empty? What if the customer wants to order more than three items? The answer to this problem is to add a new table to the ER diagram. This new table is called a junction table or a bridge table. A junction table or bridge table will resolve the problem associated with many-to-many relationships in relational databases, by creating two one-to-many associations instead of one many-to-many association. In our example, the junction table is called order line. The order line table has a foreign key to the food table called food ID, and a foreign key to the order table called order ID. Each order ID in the order line table can have as many rows as required for each food ID. So if you have a problem with many-to-many association in your database design, you may need to think about how you could use a junction table to resolve that in your own ER diagram.

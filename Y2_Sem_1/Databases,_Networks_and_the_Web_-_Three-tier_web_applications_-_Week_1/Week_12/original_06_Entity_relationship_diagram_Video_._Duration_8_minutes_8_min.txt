Now this may have been quite a challenge for you. It can be very tricky to decompose a set of written requirements and translate them into specific technical requirements needed for an application. Let's look at a similar set of requirements to those presented in the lab and work through how the design might be approached. The scenario we will use is an application that allows users to create playlists from songs. Here we have some requirements for our playlist app. Users can search for songs by name or artist. Results should be sortable by song name, artist name or release date. Users should be able to click on a song to view all its details. Users can create a new playlist and add songs to it. Users should be able to view all the playlists and the songs in the playlist and only administrators can add new songs. So how can we start to translate these requirements into a clear technical design? Thinking through the user interactions can help to get an understanding of a set of requirements. The first requirement that users can search for songs by name or artist tells us that we need some sort of search page. Perhaps something like this where a song name and artist name are entered as search terms and a search button will execute the search. The requirement that results are sortable by song name, artist or release date suggests a table of results with options to sort by the table columns, song, artist and release date. The requirement that users should be able to click on a song to view all its details suggests we should add a clickable view button or some similar mechanism which then shows the song details. The requirement that users can create playlists and add songs suggests a form to create a new playlist, some mechanism to add songs to a playlist such as a button next to each song on the search results page and some way to say which playlist the song should be added to such as a form with a drop down list of playlists. The requirement for users to be able to view playlists and the songs contained in them suggests a screen showing all the playlists. A view button could list the songs in the playlist. This final requirement that administrators can add new songs suggests another type of user, an administrator and a form to add new songs allowing the song name, artist name and release date to be entered. Now let's turn our attention to the data model itself. We need to identify objects or entities and fields or attributes that will make up the tables in our database. To identify objects and fields it can be useful to highlight all the nouns in the requirements. These will usually indicate the objects and fields we need to model in our application. Here I've highlighted the nouns in blue. Some nouns will be repeated, sometimes varying from singular to plural and by the end of this process you will have highlighted a target list of objects and fields for your data model. The unique list of nouns can then be taken and organized into related terms and relations between the terms identified. Tidying this up we have the start of a data model showing the relationships between different objects and some of the fields within the objects. At this point we can start to refine the model. Each object here will need a primary key to uniquely identify a row and relate it to other objects. Artist needs an artist ID, song needs a song ID, playlist needs a playlist ID and user needs a user ID. We can then start to look at the cardinality of the relationships. What cardinality do we need between artist and song? Well we know that one artist can have many songs. Can one song have many artists? Well yes this is possible but we may make a decision to model only a single artist, the main artist, per song. These sorts of decisions often need to be made when modeling databases. There is often no absolute correct answer and it depends on the requirements of the application. So in this case we will use a one-to-many from artist to song indicated by the crow's feet symbol here. Don't forget that we need to add the artist ID to the song object to link it to the artist object. let's look at the relationship between song and playlist. A playlist can have many songs and a song can be on many playlists. So clearly we need a many-to-many relationship but as you saw in the lecture to model a many-to-many relationship in a relational database we need to introduce a junction table. Let's call it playlist song. We include foreign key references to the two tables being joined playlist and song. Finally we can take a look at the link between playlist and user. One user can have many playlists. Can one playlist have many users? Again this could be possible depending on what sort of functionality we want our application to have. If we want our application to allow multiple users to manage playlist we'll need a many-to-many relationship here. But let's assume this isn't a requirement. So we will keep this as a one-to-many adding the user ID to the playlist object. So here we have our final data model which we can use to create tables in our database. It's important to note that the process of devising such data models isn't always clear cut and linear.

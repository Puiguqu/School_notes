Remember primary keys in relational databases? In the concept of relational databases, a primary key provides a unique identifier for each record in a table. Here each row in the book table or each instance of a book in our table is identified by an ID number as the primary key. But when building relations between tables in a relational database, we also need another key which is called a foreign key. A foreign key in a table is the primary key from another table which is used to model a relation between the two tables. Going back to the newspaper article example we have discussed throughout this course, the author name filled in the article table is a foreign key. In this case, it's referring to the primary key author name in our author table. As you can see, the data are the same and relate to each other. You might remember that earlier in the course we had a discussion about whether author name is a good field to link to tables. As I hope you realize, the answer is no. It is not a good field to choose because people's names are not unique. So author name is not a suitable primary key for the author table and for the same reason it is unsuitable as a foreign key in the article table. It is better to have a unique ID for each author as the primary key in the author table and then use this as a foreign key in the article table. In this example, the name author ID refers to the ID field in our author table. So author ID is the foreign key in the article table. Let's look at another example based on the bookshop database you have created in your lab exercises. At the moment you should have columns for the book table which is ID, the title or name, and the price and the primary key is the ID. Now imagine you would like to add a publisher field to your book table. Instead of repeating all data related to each publisher in each record of the book table which results in redundancy and modification issues as we discussed in previous topics, you create a separate table named publisher which includes columns for all the data you want and then define a column in book table as the foreign key to publisher table. In this example, you can see that there is a column in the book table that refers to the ID column in publisher. So publisher ID in book table is an example of a foreign key. You will have the chance to implement this database in your lab exercise this week and I have provided lab instructions for you to do so. You should remember the concept of constraints in database design. The foreign key constraint ensures that referential integrity of data joining any two tables by limiting the content of the foreign key to the values of referenced primary key. This is necessary in addition to primary key constraints which we discussed earlier to ensure the accuracy and integrity of data stored in databases. Now let's look at the implementation of foreign keys in MySQL. Let's assume you are creating a new table. If there is a foreign key in your table, you need to specify it. This is in addition to the definition of the primary key. You also need to specify which field in which table this foreign key is referring to. In this example, publisher ID refers to ID field in the publisher table. Looking back again at our two-table book publisher database, now let's write an SQL query to retrieve data from this database. First let's write a query on a single table such as find name and price of all the books priced more than 30 pounds. The SQL statement for this query is select book.name, book.price from book where book.price is greater than 30. The results would be the name and price of two books namely database book price 40 pounds 25 pence and express book price 31 pounds 99 pence. Now let's write a query to retrieve the name, price, and publisher names of all the books priced more than 30 pounds. As you can see, in order to run this query, we need to retrieve data from both the book and publisher tables. As you will remember, the join statement in SQL lets us merge data from multiple tables. The SQL statement is select book.name, book.price, publisher.publisherName from book joinPublisher on book.publisherId equals publisher.id where book.price is greater than 30. The answer to this query would again be two books database book 40.25 McGill Limited and express book 31.99 Oxford Press. Please note that the publisher for book id 3 is publisher id 2 which is Oxford Press. The general format of SQL join is you have the keyword select, then field name 1, field name 2, and etc. from table 1, join table 2, on join condition, where select condition. SQL join would also let you join more than two tables.

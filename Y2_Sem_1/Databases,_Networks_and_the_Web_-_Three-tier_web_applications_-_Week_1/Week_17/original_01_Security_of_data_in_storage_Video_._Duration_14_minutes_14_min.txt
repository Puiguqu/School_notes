Here are a few you might have thought of. Hackers and cybercriminals are the threat that most often comes to mind. These are individuals or groups who target systems for financial gain, disruption, or to steal sensitive information. Malicious insiders refers to employees, contractors, or other people who have legitimate access to a computer system, but who intentionally misuse it. When thinking about insiders, we also need to think about non-malicious employees and other people who unintentionally cause problems due to the level of access that they have to the systems. Industrial espionage involves gathering information about an organization or a competitor. This could be to obtain trade secrets or other inside information. Cybercrime and sabotage can be a real threat for certain sectors. Natural disasters include floods, fires, earthquakes, etc. Hardware and power failures, even if temporary, could have a major detrimental impact on computer systems. Data leakage refers to situations where data gets out to where it shouldn't be, for example being copied onto a USB stick or sent inappropriately in emails. All of these threats map to three main risks. Data might become unavailable, either temporarily or permanently, i.e. data could be lost. Data might get corrupt, or data might get into the wrong hands. We need to think about how to protect our web applications and databases against these risks and threats. But at this point you might want to pause and think, what do you do to protect your data? So how do we go about securing our data? It helps to think about two distinct problems, securing data in storage and securing data in transit. By data in storage or data at rest, we mean data held in databases or other storage media such as backup tapes. Data in files is also included, or even data held within our code. By data in transit or data in motion, we mean data moving from one place to another over a network. Here are some examples of risks associated with data in storage. A hard disk holding our database crashes and we lose all our application data. A careless employee accidentally deletes an important customer's data. A hacker gets access to our database and steals our applications, usernames and passwords. So what do we need to think about when building our web applications in order to secure the data in storage? Well, there are many things. Physical security. We don't want our web application and database running on a computer that can be just picked up and taken away. We want to ensure the physical computer is physically secured, probably in a secure data center. Access control. This is about restricting who has access to a system and only giving users access to the parts of the system they need. Separation of duties. This is about limiting the degree of access that users have so that no one user has access to so much of a system's capability that they can cause serious damage on their own. For example, in a large company, the application developers may not be given access to production systems. Instead, a different group of technical staff will have permission to deploy new versions of software to the live environment. Encryption obscures data so that it can't easily be read if it is accessed. Reading the data would require the relevant software key to unlock the data. Data patching and updating software is a ritual that any computer user is used to. But this doesn't just apply to our personal computers. Servers and server software needs to be updated too. Backups protect us against data loss. Databases in particular should be regularly backed up. A disaster recovery plan is a process that kicks in when a disaster occurs. A disaster could be something as small as a disk failure or as big as the destruction of a data centre. Redundancy involves having copies of systems or data so that if a loss or failure occurs to one copy, another copy can take its place. This can take many forms such as duplicate hard disks or copies of complete computer systems. Let's now take a deeper look at encryption. Let's say we want to register new users on a web application. Users will provide a unique username and password. We can store all the usernames and passwords for registered users in a database. But is storing the plain password like this a good idea? No, it certainly isn't. If a hacker gets access to the database, they will have all of your user's passwords. This not only compromises your site, but other sites if the user reuses passwords. A better approach would be to use a software key to encrypt the password. This encrypted password could then be stored in the database. When a user logs in, we can retrieve the password stored in the database, de-encrypt it and compare it against the password entered by the user. So this feels much more secure than storing plain passwords. But how much more? If a hacker gets access to the database, all they see is the encrypted passwords, which can't be used to log in. So this feels pretty secure. But what if the hacker somehow gets access to the key? In this case, the encryption no longer protects us because the hacker can unencrypt all the passwords. So is there a better way? Instead of using an encryption key, we could use a hash function. Hash functions are one-way functions. We can hash the word gravity to produce this obscure hash password. But there is no way to reverse the process and get back to the original password. So when a user registers, we can create a hash password and store it in the database. Now when a user logs in, we need to hash the password they entered and compare it to the hash password stored in the database. So this is a much more secure way of storing passwords than just using an encryption key. But are there still some risks? A significant problem is the so-called dictionary attack. Readily available tables called rainbow tables list common passwords and their hashes. A hacker with access to the database and a rainbow table can search for all the stored passwords in the rainbow table. If they find a match, the password is revealed. So how can we protect against dictionary attacks? A technique called salting can help protect against dictionary attacks. Salting adds an extra random string called a salt, shown in blue here, at the end of a password before it is hashed. In this way, even users with the same password will get a different hash. The salt must be stored in the database alongside the hashed password. Now when a user logs in, the salt is retrieved from the database. The salt is added to the password that the user entered. The hash function is then applied. The generated hash is then compared to the one stored in the database, and if they match, the login is successful. Now even armed with a rainbow table, a hacker won't be able to simply look up the hashes to obtain the passwords. So this feels pretty secure, but are there still some vulnerabilities? Well, yes. Given a list of common passwords, access to the salts in the database, and a powerful computer, a hacker could run all common passwords combined with all salts in the database to see if any of the hashed passwords stored in the database match. These sorts of brute force attack become more possible as computing power gets faster. To deal with brute force attacks, a hash and salt algorithm called bcrypt was created. Bcrypt is designed to be slow. So slow that a brute force attack is too slow to be productive, but not so slow that users logging in would be inconvenienced. You can adjust how slow it is, slowing it down as hardware performance improves over time. In the labs, we will use bcrypt to encrypt the passwords. It has access to every database and other objects that MySQL holds. By giving your application root access to the database, you are giving it access to control everything about MySQL. This is too powerful for one app. So what's the solution? We can create a new database user that only has access to the My Bookshop database. This limits our application's access and control of MySQL.

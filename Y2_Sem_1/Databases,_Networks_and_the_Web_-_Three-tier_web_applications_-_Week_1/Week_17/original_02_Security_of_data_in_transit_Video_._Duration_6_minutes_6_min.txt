We also looked at some mechanisms to reduce the risk of these issues, and looked at encryption of data in storage. Let's now look at the security of data in transit. We have similar concerns about the threats and risks, but the solutions need a bit of a different approach. Here are some examples of how data in transit might be at risk. A hacker intercepts network traffic, reading the usernames and passwords being transmitted. A hacker intercepts network traffic, changing the messages exchanged between the two parties. So how can we protect data being transmitted over a network? We need to encrypt our data before transmitting it. Fortunately for us, there are existing protocols that can be used to encrypt our transmitted data, such as HTTPS, FTPS and TLS. So let's look at how encryption over a network works. Consider a scenario where two computers are connected by a network. Let's say these computers are owned by Alice and Bob. If you've studied security in computing before, you may already be familiar with Alice and Bob. Now Alice has a message she wants to send to Bob. The message will need to be sent across a network, which to all intents and purposes can be considered to be a public space. And a malicious person may be lurking in that public space, ready to intercept and read the message. They may even change the message before passing it on to the intended recipient. In order to protect against such risks, Alice may choose to encrypt the message using an encryption key. As long as Bob has a copy of the key, he can de-encrypt the message and read it. Any hacker intercepting the message won't be able to read it. This seems nice and secure, but if the hacker somehow managed to get hold of the key, he would be able to decrypt the message. So how do we solve this problem? Asymmetric encryption provides a solution. Alice and Bob both create two keys, labelling one as public and the other as private. Now the thing to note about these pairs of keys is that any key can be used to encrypt a message, but only the matching key from the pair can be used to decrypt the message. No other key can decrypt the message. So Bob gives a copy of his public key to Alice. Alice now encrypts her message using Bob's public key. By doing so, she is saying that her message is intended for Bob and only Bob. When Bob receives the message, he and only he can de-encrypt it, because he is the only one who has his private key. If a hacker intercepts the message, even if he has a copy of Bob's public key, he will be unable to read the message. Alice and Bob can share their public key with the entire world, but their messages are secure as long as they never share their private key. Asymmetric encryption is widely used as a secure way of transmitting messages. Various standard protocols exist to provide encrypted data transmission. TLS, which stands for Transport Layer Security, is the main one on which others are based. TLS is an evolution of SSL, which is more widely known due to its widespread historical use. TLS uses encryption to transmit data over a network. Asymmetric encryption is used to establish the initial secure session between the two computers. HTTPS provides the same protocol as HTTP, allowing client computers to communicate with web applications, but the data is transmitted using TLS, providing encrypted transmission. Note that you must install a TLS certificate on your website for HTTPS to work. FTPS is a secure version of FTP, which is the file transfer protocol used to transfer files from one computer to another.

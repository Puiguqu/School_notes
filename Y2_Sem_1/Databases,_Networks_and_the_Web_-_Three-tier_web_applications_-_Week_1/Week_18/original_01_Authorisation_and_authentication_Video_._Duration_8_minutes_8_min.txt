Authentication is about checking that the user is who they claim to be. Authentication is about checking whether a user should be able to access a specific resource or not. Authentication is often implemented through some form of login. Authorization is implemented through what we call access control, or restricting access to resources based on the permissions granted to a user or group of users. Access control restricts access to resources. For example, if we had a delete user capability in our application, we wouldn't want any user to be able to delete users. We would want to restrict that capability to admin users who are logged in, and who are specifically permitted to manage users. Applications will generally have some access control approach. At its simplest level, it might be a case of determining what resources a user has access to when they are logged in, versus when they are not logged in. Here the resources are represented by different shapes. When users are not logged in, they can only access the blue circle. When they are logged in, they can access the other shapes as well. In this example, logged in users are further subdivided by the type of user, with admin users having access to all resources, and general users having access to only some of them. An alternative approach is to create groups of capabilities. Users can be assigned to zero or more groups, which gives them access to system capabilities. So here the user has been given access to the HR group and the user admin group only. This determines what resources they can access. In this case, the resources represented by the diamond, hexagon, and triangle. For our bookshop app, in the next lab, you will implement authentication and authorization. You will implement a registration and login feature to provide authentication functionality. You will then restrict access to the application routes depending on the user. Users who are not logged in can access most pages, such as the homepage, list of books and search pages. But only users that are logged in will be able to access the ad book page. Now there is a challenge when it comes to authentication and authorization in web applications. HTTP is a stateless protocol. By that we mean that HTTP doesn't require the server to retain any information about the users accessing it. So any request made by a user is no different to any request made by another user. So if that's the case, how can we restrict access to certain resources on the server, such as our ad book page? The server cannot know if the request is coming from a genuine admin user, who should be able to add books, or any other user who should not. To solve this problem, we can store some information about each user. If each user is assigned a unique ID, shown here in green, and that ID is available in the user's browser as well as on the server, then any request response communication between the browser and the server can be authenticated. Now that authentication for each user can be used to authorize access to a resource such as the ad book page. Storing information for each connected user in this way is called session management. Each user logging into the site initiates a session. Cookies are a way for web servers to store small pieces of information in the user's browser. To implement session management, the web server can pass a link to the session back to the browser in a cookie. Let's take a look at how this works. When a user logs into a web application, a cookie can be generated by the web server. And that cookie can be passed back to the browser in the response. And it can be stored in the browser. When a user then makes another request, say to the list books page, the cookie stored in the browser is sent in the request to the web server. The web server can use the cookie to identify the session and provide an appropriate response such as sending back the web page requested. The cookie is passing every request made to the server, so the session can always be identified. Based on the session information, the server could choose to deny access to a resource such as here where it's not allowed access to the add book page for this user. To implement sessions in our Node.js applications, we can use a module called Express Session. You can install it by running npm install express session. And you can do this from a terminal. To use it in your code, you need to require express session. And then you need to set up a session middleware function. There are a few parameters of interest here. The secret is a string that you should set up that is only known to your web application code. It is used to encrypt and digitally sign the cookie that is passed back to the user. This prevents any issue where a cookie might be tampered with. The max age property determines how long the cookie will last in milliseconds. When the cookie expires, the session ends and the user has to log in again.

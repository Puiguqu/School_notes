Let's take a little time to understand how this code is working. When a user fills in the login form and presses the login button, a POST request is sent to the web application and is handled by the slash logged in route handler. The route handler must check the users table in the database to see if the username and password provided match a username and password already in the database. If they do, the user can be logged in. If they don't, they will see an appropriate error message. Because we are storing hash passwords in the database, we need to hash the password entered by the user before we compare it to the password stored in the database. To understand how this works, we need to dive into the code. Let's take a look at the middleware function, the slash logged in route handler, that handles the login. A query is set up to select the ID and hash password from the database for the user that matches the username entered in the form. When the query is run, the query will retrieve the row matching the provided username, iNewton in this case. The data from the database will be available in the result object, which is passed to the anonymous function, which is called when the query completes. Here we should have just one matching row, so we have a list of one object, that object containing the ID and hashed password for the matching username, iNewton. From the database results, we can extract the hashed password and the user ID. Now we have the plain text password from the request body, shown in green, and the hashed password from the database, shown in blue. We need to compare the two of them, but of course they are different, because one is hashed and the other is not. So we use the bcrit compare function. This will hash the plain text password from the request body, and compare that hash with the hashed password that was stored in the database. In this case they match, and so we can allow the user to login. If we have a successful login, we will store the user ID in the session object, which is part of the request that will be passed to all root handlers. So from now on, when we handle any root, we can check if this user ID is present. The presence of a user ID will indicate that the user has already successfully logged in. Now that we have a mechanism that checks if a user is logged in, we can protect roots that we only want logged in users to access. The adbook root is one such root. We don't want any user to add books to our bookshop site, we only want users who have logged in to be able to do so. As we just saw, the user who is logged in, will have a session object in the request containing their user ID. The code for handling the slash adbook root renders the adbook.ejs page. In order to protect this page, and prevent unauthorized access to it, i.e. access by a user who is not logged in, we need to check the user ID in the session object. If the user ID is present, we know we have a logged in user, and we can allow them to access the page. We can achieve this by adding another middleware root handler called redirect login, inserting it here, before the existing middleware root handler that renders adbook.ejs. When node.js handles a call to the slash adbook root, it will call redirect login first. Redirect login will check the user ID in the session. If it is not present, the user will be redirected to the login page. If the user ID is present in the session, node.js will call the next root handler.

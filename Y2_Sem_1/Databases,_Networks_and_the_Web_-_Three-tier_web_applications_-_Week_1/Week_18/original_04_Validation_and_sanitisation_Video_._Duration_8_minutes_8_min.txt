Take a look at the form we created to register a new user. When the form was filled in, a post request was made to the registered route on the server and the form data was inserted into the users table in the database. But consider the sorts of poor data that a user could enter into a form such as this. They may enter an invalid email or leave the username blank, or enter a ridiculously short password in our web applications. We want to protect ourselves against such poor data entry, but the problem is even worse than just having poor data. Web forms that allow unrestricted data entry are prone to a type of malicious attack called an injection attack. Let's look at two of these cross site scripting attacks and SQL injection attacks, in a cross site scripting attack, a malicious user types client side JavaScript code into a web form. The web form is processed by the middleware root handler, generating a response that contains the information that that was entered in the form. The contents of the form are then displayed in the browser, causing the malicious code to run. You can see this in action with the My Bookshop app. Fill in the form, but enter a small piece of JavaScript to display an alert. In this case, the user has entered the client side script tags with an alert showing the message got you. Under normal situations you would expect your web application to just display a friendly welcome message showing the user's name as entered in the form. But because the form contains client side JavaScript code, that code is embedded in the response HTML run by the browser. So the script alert got you is executed displaying an alert box in the browser. Now this is quite an innocent attack causing no real damage. But there are many more malicious cross site scripting attacks that can be performed. Session hijacking is where a user's cookies are stolen and used to impersonate that user. Credential theft can be achieved by injecting a fake form for the user to fill in. Key login code can be injected to listen keystroke events and send them to another website. Phishing attacks can be achieved by injecting code that redirects users to a fake site that steals their information. SQL injection is another code injection technique used by hackers. In this case, the malicious code is not JavaScript but SQL. Consider this form that is used to delete a user. Here is the code that handles the post of the form to the web server. What will the code do when the name Bobby is entered into the form, the code will build a SQL statement that deletes the user from the database. The name Bobby is inserted into the SQL statement and one user will be deleted. But what would happen if the user enters this string bobby close single quote or two single quotes equals and one single quote. In this case, the SQL statement generated will include an expression that matches the username bobby or where empty string equals empty string. This last clause of the expression is always true. Empty string always equals empty string. So the delete statement will result in all users in the users table being deleted. There are various other examples of SQL injection that could have similar catastrophic effects on our database. So we've seen a couple of injection attacks, but how do we protect our applications against such attacks and other inappropriate data inputs from users? There are two methods we will look at Validation and sanitization. Validation with Express Validator implements checks on the data that a user inputs into a form. Checks are performed on the server side, so this is not client side checking that could be performed with JavaScript or HTML. The Express Validator module can be installed by running NPM Install Express Validator from a terminal. The module can then be imported, giving a check and validation result function that can be used in your code. Now, when you implement the form handler code such as this registered route handler, you can add checks to ensure that the inputs are valid. Here, the checks in yellow ensure that the email field is a valid email, that the username is not left empty, and that the password is at least six characters long. If validation errors occur, the errors can be reported to the user. In this case, they are just returned in the response, but you could implement a more sophisticated way to show the errors. Sanitization is used to combat injection attacks. It will clean up form inputs, removing malicious scripts The Express Sanitizer module can be installed by running NPM Install Express Sanitizer from a terminal. The module can then be imported using Require and linked to Express using App Use before using the value entered in a form, run it through the Sanitize function to remove any malicious scripts. So if JavaScript code was entered into the first name field on the form, the entire script would be removed. These are just a few of the techniques you can use to combat bad and malicious data inputs in your forms.

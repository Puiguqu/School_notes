Let's take a little time to understand how this code is working. The register form we implemented before this week allowed email addresses that are invalid to be entered and stored in the database. Validation can help us protect our applications from such invalid data. The current code processes whatever data is provided in the form, inserting it into the database. We can add a validator for the email field. Here we use the isEmail function of Express Validator to check if a valid email address has been entered. An error will be generated if it hasn't. Any validation errors can be picked up with the validationResult function and handled in a suitable way. In this case we are just sending the errors back in the response so the user can see them. Multiple fields can be validated by adding them to the array of validators. Here we perform a check that the email is provided, i.e. it is not empty. Here's another check that checks that the email is correctly formatted and is a valid email address. Here's a check that checks that the username is provided because usernames are compulsory. And here's a check that the password is at least 6 characters long. The errors returned by the error handling code from Express Validator are formatted as a JavaScript object. In the lab we just displayed this to the user in the raw JSON format, but instead it could be passed to say an EJS form and displayed nicely formatted to the user. So why don't you try that for yourself as an additional exercise. Now a major risk for web applications is an injection attack. In the lab we saw a simple example where a piece of JavaScript code was entered as an input into the register form. When the form was submitted, the JavaScript code was executed by the browser causing an alert to pop up. Did you work out the mechanism by which this occurred? Let's go through that now. In main.js in the slash registered root handler, this code was executed when the registration was successful. It creates a string containing a friendly message, showing the user's name and email address and sends this message back in the response. When our malicious user entered the script in the first name field, in the code the contents of the form were extracted in the request body and the response string was generated. Now as you can see, the response string contains a JavaScript script that displays the alert. This is sent in the response back to the browser and the browser sees the script tags and treats the text inside it as JavaScript code and so it executes it, causing the alert to be displayed in the browser. So to remove the risk of a script injection like we just saw, we can use express sanitize. The value of rec.body.first, which contains the malicious script, is passed through the rec.sanitize function. This removes any scripts or suspicious characters from the input, leaving a sanitized value.

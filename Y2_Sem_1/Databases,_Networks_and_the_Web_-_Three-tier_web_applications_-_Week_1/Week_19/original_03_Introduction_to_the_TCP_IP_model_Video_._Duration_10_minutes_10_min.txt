TCP/IP or transmission control protocol and internet protocol, is an internet protocol stack that provides communication between two computers through the Internet. It is likely that you mostly work at the application layer, but you will still require some understanding of TCP/IP protocols. So it is important to have a basic mental model of the Internet protocol stack. First of all let me explain what protocols are. Protocols are a set of rules for when and how computers send and receive messages between them. It is like a language. When I start talking in English, you know the language is English, and you know that you need to use the meaning of English words called semantics, and the grammatical structure of sentences in the English language, in other words, its syntax to understand me. Essentially, semantics and syntax are parts of grammar, which is the word we use for the rules of a language. If our communication was a two-way communication, which unfortunately is not, you would need to use the same set of rules to respond to me so that I could understand you. Protocols are the same. When two computers talk to each other with an Internet protocol stacks such as TCP/IP, the content of the message and how they send and receive are determined by rules set by TCP/IP. Both the sender and the receiver use this suite of protocols to understand each other. There are public domain protocols such as those defined and request for comments or RFCs. RFCs are publicly available documents. A protocol is tested by a technology community and then they invite others to comment on it. This allows for interoperability. HTTP, which we discussed in earlier topics, is an example of a public domain protocol. There is a second category of protocols called proprietary protocols, these include protocols such as Skype, which are not publicly available. OSI or Open Systems Interconnection is an abstract, conceptual reference model. The International Standardization Organization, ISO, set this guideline. It shows layered communication and computer network protocol design as illustrated here. There are seven layers in OSI reference model, namely: application, presentation, session, transport, data link, and physical layers. TCP/IP, on the other hand, is a standard Internet protocol stack which is standardized by the Internet Engineering Task Force, IETF. The Internet is developed around this protocol. So OSI is a theoretical model, and TCP/IP can be considered as an implementation of the OSI reference model. The OSI model has seven layers, and TCP/IP has five layers. TCP/IP is a collection of protocols, and I will explain each layer of the TCP/IP model one by one later in this topic. Briefly though, the application layer supports network applications. The transport layer provides a process-to-process data transfer. The Internet or network layer is where the routing of datagrams from source to destination happens. The link layer provides data transfer between neighboring network elements. Finally, the physical layer is bits on the wire. In both models, all layers, except the physical layer, are software. The physical layer is hardware, such as wires, Wi-Fi, optical fiber cables and so on. This part of the layered architecture is responsible for the actual transfer of data between two computers. This is how the theoretical OSI layers corresponds to the real layers of the TCP/IP model. The application, presentation, and session layers in the OSI reference model correspond to the application layer in TCP/IP model. The transport layer is the same in both models, as is the network layer, which is sometimes called the Internet layer. The data link layer is the link layer, and the physical layer is again the same in both models. Now let's look at example protocols in each layer of the TCP/IP model. A collection of protocols is called an Internet protocol stack. HTTP, FTP, SMTP, SSH, DNS, and Telnet are application layer protocols. TCP and UDP are transport layer protocols. Internet protocol corresponds to the Internet or network layer. Ethernet, wireless, and ADSL are protocols in the link layer. Before explaining each layer in the TCP/IP model, let me explain why computer networking is designed in this layered way, using the analogy of sending a parcel. Imagine you would like to send a parcel to a friend who lives in a different city. You write the sender's and receiver's addresses on the parcel clearly distinguished from one another. Then you go to the post office, buy a stamp, and stick the stamp to your parcel before putting it into a post box for collection. Then the postal service collects all the letters and parcels and puts them into different categories for various destinations. All transport services such as land, trains, or even airplane are then used to transfer the parcel to the destination's local post office, which will then send a person to deliver the parcel to the ultimate destination, your friend's address. So as you can see, there are many different people and services involved at each stage in the sending of a physical parcel. Each stage, or shall we say layer, is responsible for certain jobs, receiving the parcel from the previous party, doing some work on it, and then passing it to the next stage or layer. Dividing the responsibilities into different layers is a good way to manage a job. Furthermore, this process doesn't only involve the actual physical transfer of the parcel. Before the transfer happens, there are other administrative tasks too, such as reading the address and sorting the parcel. In the layered communication of a network, there are also different layers involved. Each layer receives data from the previous layer and does some work on the data before passing it to the next layer. Again, many admin tasks are happening at various layers. This is how a message from a sender travels to the destination. As you can see, we can think of this journey a bit like a U-shape. The message travels down through the application, transport, network, and link layers of the sender's TCP/IP, right to the physical layer. It travels along the physical hardware and then arrives at the physical layer of the destination computer, and then travels back up through the destination computer's link, network, transport, and application layers. Finally, the original message arrives at the destination's application layer. As well as passing the message on, each layer in the source adds some data as the headers to the original message. The transport layer adds a header, let's call it header t. Then the network layer adds the IP addressing headers, which we'll call header n. Then the link layer adds some data which we'll call header l. When the data arrives at the destination, each layer starting from the link layer, removes the header related to that layer before passing the message to the layer above, until the original data arrives at the application layer of the destination computer. Note that we use different terms to describe the method at each of these stages. What we call the message or data in the application layer is called a segment in transport layer. Then in the network layer it is called a datagram, and in the link layer, it is called a frame.

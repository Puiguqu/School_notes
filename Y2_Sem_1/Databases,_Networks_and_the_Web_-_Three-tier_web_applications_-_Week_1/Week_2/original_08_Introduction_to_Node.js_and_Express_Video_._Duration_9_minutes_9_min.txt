First, it sends the request to the web server. Then, it waits until the database access and querying is finished. If a new request comes in from another client while the first request is waiting, another thread is allocated to handle that new request. The current threat cannot be allocated to another request. Once the querying is done, PHP returns the database records it has found to the client browser. So now PHP is ready to handle the next request. Let's imagine the same request with Node.js. Well, if a new request comes in from another client, the same thread is allocated to handle the second request. When the database has found answer to either query, the server returns the database record found to the client web browser. This is because the asynchronous nature of Node.js is suitable for handling I/O-intensive applications and it is highly scalable. Examples of app like this with the e-commerce apps. But when should you not to use Node.js? It is not advisable to use Node.js for CPU intensive applications, like video encoding and image manipulation services. This is because in this type of app the bottleneck is CPU usage, not input output. So it is better to use a web development environments that are multithreated in these cases.

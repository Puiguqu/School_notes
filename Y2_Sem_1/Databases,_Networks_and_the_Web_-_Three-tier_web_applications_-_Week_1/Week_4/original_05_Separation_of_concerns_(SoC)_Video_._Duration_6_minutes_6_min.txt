First, I'll explain what SoC is. We need to know what we mean by a concern, and then what the benefits of SoC are. As always, there are a slightly different definitions out there. As the technology training website oreilly puts it SoC is the idea that each module or layer in an application should only be responsible for one thing and should not contain code that deals with other things. Wikipedia says, in computer science, separation of concerns or SoC is a design principle for separating a computer program into distinct sections, such that each section addresses a separate concern. But what is a concern in this context? A concern could potentially be anything that affects the computer program. So that could be presenting contents to users or database connections or a piece of logic in your code. Now the question is, how can you achieve SoC in your code? The main idea of SoC is to decompose the code into parts with minimum functionality overlap. So first, we need to identify those parts or concerns. And then remove dependencies as much as possible. One way to achieve SoC is modularity, employing modules for the construction of your application. And how can you achieve modularity? By hiding information inside the module and providing well-defined interfaces for the interaction of the module with outside world. This is called encapsulation. Therefore, a modular program is a good example of a program that has applied the SoC design principle. Another way to achieve SoC is the layering of functionality on the application. Three-tier applications and MVC, or model view controller, software design patterns are examples of layering functionalities. SoC is about planning logical layers, and then designing each layer in a way that it is loosely coupled by interfaces with other layers. What do I mean by loose coupling? It means that you have to implement each layer independently from the others. Also, you need to remove dependencies between various parts of your application as much as possible. It is, of course, not possible to remove all dependencies, because software is a whole and all sections need to relate to each other to bring functionalities to end users. So instead of talking about removing all the dependencies, we talk about loosely coupling them. As I said before, a layer designs and information systems are another way to ensure separation of concerns. Examples of layers could be the presentation layer, the business logic layer, or the database interaction layer, to name just a few. In fact, SoC can be achieved in different levels. It could be expressed at the level of functions in your code, or modules, or even controls such as the GUI or graphical user interface, controls and widgets, and sometimes tiers and services. So let's get to the main point. What are the benefits of SoC? First of all, SoC reduces complexity by breaking a large application down into many smaller units of encapsulated functionality. Secondly, it helps the DRY, or don't repeat yourself, coding strategy. SoC makes it easier to keep things separate. The DRY coding strategy is about not repeating pieces of code in your program design. Instead, you rename them and reuse them. Thirdly, SoC also improved the portability of the application. Hardware diversity is increasing constantly, so portability and software adaptation are real necessities nowadays. SoC facilitates portability because concerns are separated, and software reuse is possible. Plus, we can reuse pieces of code and add new modules for new hardware. Lastly, SoC improves the maintainability and testability of an application. When you have several almost separate parts instead of a fully integrated piece of software, it is easier to test spot errors, and find and fix bugs. It's a similar case for other maintenance issues. So generally, it's much easier to test and maintain software when the SoC design principle has been used. in this topic, you will have a discussion based on these questions.

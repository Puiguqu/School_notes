As you know HTTP stands for hypertext transfer protocol. It is a protocol to access data on the world wide web. HTTP is a client-server application protocol and it sets the rules of interaction between a client program or a browser and a server program or a web server. Then traction of client and server happens by exchanging requests and responses. As we saw in the steps, I outlined earlier the client or web browser makes a request for a particular resource. On the other side server program or web server receives the request, the server runs continuously constantly alert for such requests coming in and ready to handle them when they do. You sometimes said that the server is listening. HTTP defines a set of methods for making these requests. Each request method is designed to perform a specific action on a given resource. HTTP request methods are sometimes called verbs and include get, put, post, delete, patch, head, connect, options and trace. I will only discuss get and post in this course, but you might like to look into the others for your own interest. Get and post are two of the most common HTTP methods. GET is used to request data from a specified resources on the server. POST is used to send data in order to create or update a resource on the server. So let's look at these two a bit more closely. So as I said the GET method asks to retrieve data from server. Requests using get should therefore only retrieve data and should not modify data. GET is not suitable when dealing with sensitive data as it is not secure. That is because the data query string including name and value parse is sent within the URL of a GET request and can therefore be seen by others. POST is used to send data often from an HTML form to create or update a resource on the server. The data is a stored in the body of the POST request not in the URL. I mentioned HTML forms here. So let me tell you a bit more about them now. Forms are used in dynamic web applications to collect users input data. A web form may contain any number of input fields including type such as edit boxes, checkboxes, drop-down menus, radio buttons, and so on. You're probably familiar with this from online shopping or electronic surveys and things like that. This information is then sent to the server as an HTTP request. So, this is the first step in user interaction with databases through web applications. Display a form and send an HTTP request to the server by the GET or POST methods. Now the question for you is when you design a form in your dynamic web application to collect users inputs, which HTTP request method should you use, GET or POST? To help you make this decision, let me explain the difference between GET and POST by using the analogy of a person buying a sandwich in a cafe. Here he is. First the customer tells the cafe owner that he'd like a sandwich. Then he will probably be asked what sort of sandwich he would like. In other words, what do you want in it? This part of the ordering process is similar to when a web client request some data and the server ask what do you want? The data resource is the sandwich but so far we don't know what we need from that resource or to use our analogy what type of sandwich it should be. The customer replies, I'd like a sandwich with chicken in it. In our analogy, as the customer describes the sandwich filling he wants the client is describing the data it wants. This is all done openly in the cafe. This is similar to HTTP GET request in which the specification of the resource we would like to retrieve is sent in the URL as a query string. Both the request itself and the description of the data are openly stated. On the other hand, imagine the next step in the same scenario when the customer needs to pay for the sandwich. He will be asked, how are you paying? And in our scenario he chooses to pay by card. When we pay by card we send the data about our bank or credit account in a more secure way than openly discussing that information in the cafe. This is similar to when you use the POST method to send HTML form data to the server. Data is not sent in the URL, which is not a secure way to send sensitive data. Instead it is sent in the body of the POST request. So, the difference you should consider when choosing between GET and POST is what the data is. If it is sensitive at all or likely to be you should not use GET to retrieve it. The second criterion is that when you are creating or modifying data, you should always use POST because GET is only used to retrieve data as I said before. So far, I have explained that we can use GET and POST request methods in HTTP to ask to retrieve data from a server or to modify or create resources on the server based on the form data that is filled in by users.

Before this video, in our discussion activity, I asked you to imagine that you are employed by a newspaper to design and implement a database for archiving newspapers. This should help you to think about databases in a practical way. The question for you in this scenario is, how do you store data, say an article of a newspaper? Do you think storing a picture or an article is a good idea? Is storing the data the only purpose of a database? What if you'd like to do a search within all the articles based on a search keyboard? So can we say that querying is another purpose of creating database? The first purpose of a database is, of course, to store data. If data is not stored, we can't really do anything with it at all. But as I hinted just now, a second reason we recreate databases is to be able to access and query the data in the database. Querying means reading data from the database. That said, accessing a database is not only for the purpose of reading it. Sometimes we want to access the database in order to create, modify, or delete data. Creating, reading, updating, and deleting data are database operations that we would usually like to be able to perform. Sometimes the acronym CRUD is used for these four database operations. That might help you to remember them. In addition to the two reasons we've already talked about, a third purpose of a database is to provide an organizational structure for data. Storing it is all very well, but it is much more useful if the information is stored in a logical and meaningful way. Before we carry on, let's clarify what we mean by a database and a database management system or DBMS. A database is the data itself, collected together. The DBMS is the software that provide access to and manages that data. That's important to understand that not any collection of interrelated data is a database. As computer programmers, we have many forms of collections of data in our programs and not all of them are called a database. As Silberschatz, Korth, and Sudershan tell us, when we're talking about collections of data referred to as databases, we mean collections of data that are highly valuable or relatively large or accessed by multiple users and applications often at the same time. These authors also make the important point that the primary goal of a DBMS is to provide a way to store and retrieve database information that is both convenient and efficient. So now that we understand what databases and DBMS are, let's go back to thinking about how we use them. These days, you can see databases in almost all application domains. For example, on this course, there are databases about student registration, progress, fees, test results, and so on. Other obvious examples of database applications you might often use are social media sites and online retailers which have databases to do with users, products, trends, and so much more. There are lots of examples of application domains in your textbook. So make sure that you find time to do that reading. The way databases function nowadays can be very sophisticated. But as I mentioned at the start of this topic before databases were developed, data was stored in lists and we need to understand more about that to have a firm grasp of database design.

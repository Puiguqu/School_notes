Let's take a little time to understand how this is working and explore a bit further. When you installed MySQL on your computer, it was set up as a service, which means it runs in the background on your computer and doesn't have a user interface. Note that you'll often see this symbol, a cylinder, used to represent a database. It shows a hard disk, which is a reminder that the database will be storing data permanently, meaning that the data will still be there when we switch our computers off. Now in order to interact with the database, we need to use some form of interface. In the lab, we used the MySQL shell for this purpose. This allows you to issue commands to the MySQL service and get responses back. MySQL holds a number of databases. When you install MySQL, there are already some databases installed. These are Information Schema, MySQL, Performance Schema, and Sys. These are just system databases used by MySQL to manage itself. You can ignore them for now, but at some point in your exploration of MySQL, you might want to use these to query MySQL to understand how it is organizing its information. So when you issue the show databases command, you will see that MySQL informs you that these databases exist by listing them out. Now let's look at what happened when you issued the create database command. When we issue the command create database mybookshop, MySQL responded to this by creating a new database alongside the system database we saw before. Here we see the mybookshop database has been added. Having created the database, we want to start adding data to it, because the whole point of a database is to store data. Before we can add data, we need to select the active database. We do this by issuing the use command with the name of the database, mybookshop, that we want to use. The mybookshop database now becomes our active database, indicated by this thicker line. Now any database level commands we issue will apply to this database only. we issued the create table command. Here we specified the name of the table, books, and the columns, ID, name, and price that we want in the table, including their data types and any attributes. Finally, we specified which columns constitute the primary key for the table. This creates a table within the mybookshop database, and the table will have the same three columns, ID, name, and price, with the data types, attributes, and constraints we specified in the create table command. Notice that each column has a data type, shown here as int varchar 50 and decimal 52. Any data that matches the specified type can be inserted into that column. Notice also that the ID column has certain attributes and constraints applied. In this case, the auto increment attribute tells MySQL to automatically assign the next integer value available when the data is inserted into the table. The primary key constraint tells MySQL that the ID column is the primary key, so uniquely identifies a row in the table. At this point we can use the show tables command to see what tables we have in the database. At the moment we just have one, books, which is listed in response to the command. We can also use the describe command to ask MySQL to show the table structure that it holds for the books table. MySQL responds by listing the rows and columns in the books table. Let's now see what happens when we insert data into the books table. Here we issue the command insert into books, asking MySQL to insert three rows into the table and specifying the name and price of each book. Notice that we don't specify the ID because the ID column is defined using the auto increment attribute. MySQL will automatically assign a new ID to each row. And this is the result. The three new rows have been added and the ID column has been automatically assigned a sequence of numbers. Note that because the ID is a primary key, the ID must uniquely identify a row in the table. You can see that in this case it does, with the first book getting ID one, the second ID two, etc. Running another insert command will add a new row to the table with a new unique ID. Note in this case I didn't specify the price, so MySQL has left it blank. Unlike the ID field, where we directed MySQL to automatically assign an ID by specifying the auto increment attribute, MySQL can't automatically assign a price based on the information we have given it. Now if we insert another book, this time specifying all three columns, then all three columns will be inserted using the value specified. In this case the new row has an ID of six, because that is what we requested. When we add another new row, if we don't specify the ID, it will take the next value in the sequence, as directed by the auto increment attribute. Now say we are trying to insert a new row into the books table. Here we are trying to insert a new book with an ID of three. What will happen? Well in this case we will get an error, stating that there is a duplicate entry of three for the new book, because there already exists a book with an ID of three. So the insert statement has failed, and the books table remains unchanged. What about this case? Here we are inserting a book with the name Programming with Python, and a price £35.80. Will this insert statement work? No it doesn't. We get an error, because MySQL is expecting a decimal value for the price, but it got a string. As you can see, MySQL is very strict about the data you insert into the tables. This strict enforcement is designed to ensure the integrity of your data, so we don't end up with a mess of inconsistent or incomplete data. Finally let's take a look at the select statement. Now that we have data in our database table, we want to be able to write queries to retrieve that data. If we select star from books, we get all the rows and columns back from the database. The star means all columns. If we use the limit clause, in this case limit to, we get a limited set of rows back, in this case just the first two rows. Instead of star, we can list the specific columns we want to see returned, in this case name and price. And we will get back just the name and price columns for all books. We can also use the where clause to restrict the result set to just matching rows. In this case, where id equals 2 means we only get back the books where the id is 2.

But we haven't yet talked about the programming technique from which they come; object-orientation. To really understand object orientation, we need to understand the issue it was designed to address. Spaghetti code is the term that we use to describe code that has become so complex that even it's designer is unable to read or understand it. Say for example, we have a program with two functions, and one function calls the other function. That's the situation we could quite happily deal with. Now, say for example there is a third function and that function is called by the second function. So far so good. We could even continue this pattern to deal with a fourth function. However, when different functions stop cooling each other things can become a little bit tricky to follow. The situation gets worse when we add variables into the picture because all of the functions might be affecting those variables in different ways. So this results in a situation where a seemingly minor change can have large consequences cascading right through your program. Object-orientation deals with the situation through two principles. The first is encapsulation. Through encapsulation, related functions and variables are grouped together into entities called objects. As you already know, variables inside objects are called properties, and the functions are called methods. Well, encapsulated objects only modify their own properties. They might refer to global variables but they avoid modifying them. Objects interact with each other by making reference to each others properties and by calling each other's methods. The methods and properties of an object are called its interface. In order to prevent spaghetti code from developing, interface design follows a principle called abstraction. This is where only the essential elements of the objects are made available to other objects. The internal workings are kept hidden from the rest of the program. Well, abstracted objects have interfaces that are simple and reveal as little about the inner workings as possible. People often refer to interfaces with these qualities as black boxes. Examples of abstraction are all around us. One example is your mobile phone. The interface offers us only the relevant handles for using the device. Internal details such as the signal channel are hidden from the user. P5.js is a good example of abstraction as well. We call the rect function, but the details of how the rectangle is actually rendered to the screen is hidden from us. The net result of these two principles is to reduce the impact of changes that you make to your program.

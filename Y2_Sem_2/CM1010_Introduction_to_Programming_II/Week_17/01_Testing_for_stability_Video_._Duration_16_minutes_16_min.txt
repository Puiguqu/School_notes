This module introduces fundamental ideas about software testing, complementing previous discussions on debugging. Testing and debugging are distinct: debugging is a continuous, ad hoc process during coding, while testing occurs at a formal stage in the development lifecycle. During testing, the developer does not modify code between test cases; instead, all tests are completed, followed by a correction period. Testing requires a formalized test plan, whereas debugging does not, and testing can be performed using automated tools. Testing types are categorized along the dimension of the test level within the development cycle. Unit testing examines the smallest logical program units, such as methods, functions, and constructors, ensuring expected output for specific inputs. These tests must cover a range of both likely valid and invalid input values. For JavaScript unit testing, libraries like Mocha.js, QUnit, and Jasmine are often employed. Integration testing validates how these individual units interact when passed as inputs to one another. System testing checks the completely integrated system's functionality, often deriving test cases from formal system requirements. Acceptance testing determines if the software is appropriate for the tasks it was designed to solve, potentially including user testing.

Alternatively, testing is characterized by its methodology: black box versus white box. Black box testing evaluates the system based on external functionality, without knowing the internal code structure, focusing only on required input and expected output. White box testing, or clear box testing, focuses internally on the code structure, attempting to iterate over all logical paths through the code. Due to the exponential nature of paths with nested conditionals and loops, testing must be selective, often focusing on boundary conditions (highest and lowest acceptable values). Gray box testing is a hybrid approach that attempts to maximize the utility of both black and white box techniques. Though theory suggests unit/integration testing uses white box methods and system/acceptance uses black box methods, practice often blurs this distinction. The practical execution of a system test case, such as testing a drawing application, demonstrates the necessity of covering varied scenarios. Error handling is demonstrated by testing inputs like negative numbers or non-numeric characters, which should trigger specific error messages. Furthermore, testing should validate the state management, ensuring that when a user changes a tool (e.g., size), the change persists correctly across subsequent functions. The ability to pass the drawing tool state (e.g., the current size) across different functions confirms that the application maintains state integrity. These documented tests ensure the software meets functional requirements and handles invalid data gracefully.

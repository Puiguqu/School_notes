# Splitting across multiple files

The process of developing large-scale applications requires robust code organization to maintain efficiency and readability. When creating complex p5.js projects, a single sketch file can quickly become overwhelming, potentially accumulating hundreds of thousands of lines of code. To counteract this monolithic structure, developers must adopt the practice of modularization, thereby splitting the code into multiple, specialized files. This process is essential for keeping the sketch file terse and allowing the object-oriented detail to reside in external components. A critical best practice guideline is to assign each major constructor function its own dedicated JavaScript file. For instance, if a program utilizes both an `Emitter` object and individual `Particle` objects, these distinct functionalities should be separated into corresponding files. Following this structure, the entire constructor function for the emitter is extracted from the main sketch file. This function is then pasted into a new, separate file that must be saved as `Emitter.js`. It is crucial that the filename matches the name of the constructor function to maintain logical consistency. Furthermore, the full `.js` extension must be included in the filename for the file to be correctly recognized as JavaScript. Similarly, the separate particle constructor function is extracted and placed into its own file, saved as `Particle.js`. After this reorganization, the original sketch file becomes significantly shorter and easier to understand. However, simply splitting the code is insufficient; the main sketch file must now be updated to reference these external modules. This requires embedding new `<script>` tags into the main HTML document, similar to the existing sketch script tag. These new script tags must link to `Emitter.js` and `Particle.js`, ensuring the external constructors are loaded into the environment. There is a critical technical dependency regarding the order of script loading. The sequence must begin with the main p5 library inclusion. Next, all defined constructor files must be linked in the correct order. Following the constructor files, the primary `sketch.js` file must be placed last. This specific loading sequence ensures that when the program reaches `sketch.js`, all necessary components and constructors are already defined and available for use. By adhering to this modular workflow, developers create a codebase that is highly structured and maintainable. This strategy significantly improves collaboration and debugging efforts in complex object-oriented programs.

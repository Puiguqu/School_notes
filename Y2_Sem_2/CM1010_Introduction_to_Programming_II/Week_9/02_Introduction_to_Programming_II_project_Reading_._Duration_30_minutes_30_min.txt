The core assessment for this module requires students to extend one of three provided template applications using their own tools or visualizations. Students must select one of the three designated template downloads: `draw-app-masterV2 ZIP File`, `musicVis-master ZIP File`, or `data-vis ZIP File`. These template versions are designed to be clean, excluding the incompletions and issues found in the earlier case study version. Over the coming topics, example extensions will be provided for each application to serve as both assistance and a starting point for development. A crucial technical detail is that most demonstrated extensions will initially be implemented outside the template and must therefore be adapted to function within the established application structure. Students are expected not only to implement these extensions but also to improve upon them by including additional features based on their own creative ideas. Regarding complexity, a student may only include a maximum of two extensions originating from Topics 5, 6, or 7 in their final submission. The total number of extensions included must be determined by their respective complexity and the programming techniques utilized. Careful consultation of the mark scheme is mandatory to ensure the inclusion of all required techniques, both those shown and those written by the student. The provided example projects, such as the "Introduction to Programming II: Example final projects.pdf," should be used as guides for the scope of the final work.

Strict adherence to academic guidelines, specifically the `CM1010_Assesment_Guidelines.pdf`, is non-negotiable to ensure fair marking and prevent accidental plagiarism. Students are strictly forbidden from submitting any code that is identical to that of another student. Any sourced code obtained from the internet must be explicitly and clearly labeled in both the code itself and in the appropriate section of the written report. Furthermore, code must not be hosted on publicly available version control repositories, such as GitHub, before or after the submission period. Failure to comply with these rules constitutes an academic misconduct investigation, guided by the University of London's General Regulations. Technical constraints mandate that the code must not rely on any external servers, whether self-hosted or provided by a third party. The application must be testable solely by opening the code in Brackets and launching the program. If an external library is utilized, only a local version of the code is permissible; web links to CDN services are forbidden.

The project deliverables are divided into two parts: a midterm assignment set in Topic 6, and the final submission in Topic 10. The Midterm assignment requires the submission of the current project code, along with written responses to the report prompts housed in the ‘My Assignment’ tab. This stage's purpose is to confirm the project's adherence to main module requirements and verify sufficient progress. The assessment of the midterm project will weigh heavily on the demonstrated use of design and planning techniques, alongside the quality of written work. The Final submission requires the submission of the finalized code and a further short report document. This report must outline the project, discuss the effectiveness of the plan employed, and include a short evaluation of the code base. Students will be ultimately assessed on the quality and complexity of the code, its overall stability, and the depth of the answers provided in the final report.

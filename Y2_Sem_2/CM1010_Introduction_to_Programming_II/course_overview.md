The material provides a comprehensive review of object-oriented programming (OOP) concepts, building upon foundational principles. Initially, the process began by drawing basic shapes and utilizing numbers to describe simple attributes, such as position and dimension. Progressed learning involved the use of variables, which are employed to store primitive values. These primitive types include numbers, strings, and boolean values. Storing values in variables significantly increases the developer's control over the sketches being created. As the complexity of the created items increased, it became more manageable to group these individual variables together into sophisticated objects.

A fundamental rule of thumb in object design states that an object should always be used to describe a single, singular entity within the program. Formerly simple standalone variables are elevated within the structure and become properties of a dedicated object. By utilizing these properties, developers gain the ability to create variation and complexity within their system. Subsequently, the need arose for objects to exhibit active behaviors, leading to the incorporation of methods. These methods function similarly to traditional functions, allowing objects to perform specific actions.

Crucially, when writing a method that needs to reference the object itself, a special keyword must be employed. This keyword is `this`, which serves as an essential reference meaning "me," "my," or "I." To generate multiple independent versions of these structured objects, developers utilize specialized constructor functions. These constructors are paired with the `new` keyword to successfully instantiate many object instances on the fly.

The course structure outlines a modular progression through OOP practices. Module 1.2 focuses heavily on revisiting core object concepts and practicing their implementation. ### 12_Object_orientation_Video_._Duration_3_minutes_3_min

# Object orientation
Video•
. Duration: 3 minutes
3 min - Summary

URL: https://www.coursera.org/learn/uol-introduction-to-programming-2/lecture/tqF5m/object-orientation

Object-orientation (OO) is a programming technique designed to address the systemic issue known as "spaghetti code." Spaghetti code describes programs that become so complex that even their original designers struggle to understand or modify them. This complexity is exacerbated when numerous functions call one another, particularly when multiple functions affect shared variables in varied ways. Object-orientation mitigates this risk by employing two core principles: encapsulation and abstraction. Encapsulation groups related functions and variables into discrete, self-contained units called objects. Within these objects, variables are defined as properties, and the functions themselves are referred to as methods. Critically, encapsulated objects are designed to modify only their own properties, actively avoiding the modification of global variables. Objects interact safely by making references to each other's properties and by calling each other's defined methods. The publicly accessible functions and data of an object are collectively known as its interface. To prevent the recurrence of spaghetti code, interface design must adhere to a principle called abstraction. Abstraction dictates that only the essential elements of the object are made available to other parts of the program. The internal mechanics and workings of the object remain hidden from the rest of the system. Abstracted objects possess simple interfaces that reveal as little about the inner workings as possible. These interfaces are often compared to "black boxes" for user understanding. Practical examples of abstraction include a mobile phone, which hides complex details like signal channeling from the user. The P5.js function `rect` provides a clear coding example, as the details of rectangle rendering are hidden from the programmer. The combination of good encapsulation and good abstraction significantly reduces the impact of program changes. This architectural strength allows the object to be expanded or adapted without causing cascading failures throughout the existing code base.

To move from $O(n^2)$ to a more manageable complexity, spatial data structures and algorithms are necessary.

*   **Pairwise Calculation:** The brute-force method of checking every particle against every other particle.
*   **Spatial Partitioning (The Solution):** Instead of checking every pair, the space must be broken down into smaller, manageable zones.
    *   **Grid/Bucketing:** Dividing the simulation area into a grid. Particles only need to interact with particles in their own cell or neighboring cells. This drastically reduces the number of pairs that need checking.
    *   **Potential Algorithms:** The speaker mentions algorithms designed for this, suggesting knowledge of techniques like **Barnes-Hut** or **Octrees**, which are standard tools for fast, approximate N-body simulations.

### 3. Debugging & Analysis Techniques
The speaker demonstrates how to systematically test and confirm performance improvements.

*   **Benchmarking:** Measuring performance at increasing scales ($n=100, n=1000, n=10000$).
*   **Profiling:** Identifying exactly *where* the time is being spent in the code (the inner loops).
*   **Systematic Testing:** Confirming that optimization works by observing the runtime scaling against the expected complexity curve.

### 4. Advanced Data Structures & Theory
The discussion touches on the theoretical foundations of efficient computation.

*   **Complexity Theory:** Understanding the difference between polynomial time, linear time, and exponential time growth.
*   **Spatial Indexing:** Using data structures to map physical space onto computational data structures for fast querying.

3.  **Implement and Validate:** Apply the optimization, measure the speedup (e.g., achieving near $O(n \log n)$ or $O(n)$ time complexity), and confirm it scales properly.

Usability testing shifts the focus from mere functional operation to evaluating how effectively an application is designed for its human user and the underlying system. This process ensures the application is usable, that all controls are visible, and that the user workflow maintains a logical order. The core objective is to uncover how the application communicates its functionality and whether users can efficiently find necessary parts without disrupting their thought process. Usability testing must also assess if the app meets broader human goals, such as enabling an artist to draw desired images or accurately representing data in visualizations. To achieve this, testing is often structured around a comprehensive user experiment where potential users complete predefined tasks and give feedback. Collecting data requires identifying both quantitative and qualitative measurements. Quantitative data includes metrics such as task completion time, the total number of errors made, the number of successfully completed tasks, and overall user satisfaction scores. Qualitative data captures nuanced information about the user's experience and frustrations during interaction. A primary technique for generating detailed qualitative insight is the think-aloud protocol, requiring the user to verbalize their thoughts and feelings while using the app. Complementing this is conducting post-task interviews to discuss the participants' general experiences with the application. While there is no statistically defined number, small, representative groups of users are critical for uncovering most usability issues. It is paramount that the selected users accurately represent the target audience, as different profiles (e.g., art students versus computer scientists) will experience the application uniquely. Before testing, a detailed structure must be designed to ensure consistency across all user sessions. This structure involves creating a clear, step-by-step path using as much of the application's functionality as possible. Researchers must also standardize the initial user experience, detailing if a demo will be provided and specifying the input devices used (e.g., touchpad versus mouse). Consistency is also required in the physical environment, such as where printed instructions are placed relative to the user. Data collection expands beyond simple observation to include precise timing and counting specific errors. Users can also complete short surveys immediately following task completion to measure satisfaction. Upon collection, the data must be analyzed to determine areas for improvement and to make evaluative judgments on the application's success. A key analytical technique is looking for convergence: finding common problems across multiple users suggests a systemic error that requires immediate fixing. Furthermore, comparing task times can help pinpoint structural bottlenecks within the application's design. Although the assignment does not require iterative development, the final report must thoroughly discuss the testing process, findings, and potential improvements. Crucially, the report must include an appendix containing the raw collected data, notes, timings, or survey responses.

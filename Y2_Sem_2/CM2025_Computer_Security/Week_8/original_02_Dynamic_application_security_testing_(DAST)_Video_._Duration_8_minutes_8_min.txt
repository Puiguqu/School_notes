Some screen readers may require using CTRL in conjunction with the alt key Welcome to this session on Dynamic Application Security Testing, or DAST. Today, we'll learn how DAST helps identify vulnerability by testing on running applications. We'll also compare black-box and grey-box testing approaches, which are widely used in security assessment and DevSecure apps practices. By the end of this session, you'll understand where DAST fits into the development lifecycle and how it supports secure applications. DAST is a method of testing applications while they are running, just like an attacker would. It mimics real-world attacks by interacting with the app from the outside by sending input, observing the result, and detecting potential security flaws. DAST is great for identifying issues such as misconfiguration, broken authentications, or response handling errors. One major benefit is that DAST does not require access to source code, making it useful for testing third-party applications or staging environments. How DAST works? Imagine we are testing a web application with a login form. We develop the app in a test environment and run a DAST scanner like OWASP-ZAP. The scanner automatically enters suspicious input into the form. For example, it tries to bypass login using SQL injection or insert script tag to test for cross-site scripting such as admin or 1 equals 1 script alert XSS script to look for abnormal or unsafe behavior. If the application responds with errors, redirects or has a strange output, the scanner detects it as a possible injection or XSS vulnerability and we get a report showing what the scanner has found. From there, the development team can fix the issue before the app goes live. This is how DAST helps catch real-world issues that would be missed in static analysis. So, what is the difference between DAST and SAST? DAST is running in the application state, whereas SAST uses source code only. DAST does not need access to code. SAST does need access. DAST's testing focuses on behavior during execution. SAST's testing focuses on code logic and structure. DAST's user perspective is external from the view of the attacker. SAST's user perspective is internal from the view of the developer. Blackbox testing involves testing an application with no access to the source code, database, or internal logic. The tester behaves like an external attacker, interacting only through the user interface or APIs. This approach focuses entirely on what the app exposes to the outside world. Blackbox testing is ideal when you want to test security from the attacker perspective, making it natural for DAST and penetration testing. What is greybox testing? Greybox testing is a mix between blackbox and whitebox methods. The tester knows some information about the system, such as the type of technology used, or how user roles are structured, and this allows them to target their tests more effectively, without needing full access to the source code. It's efficient and realistic, and often used in dev-secure ops workflows to catch security flaws quickly without full audits. Advantages of DAST There are several reasons why DAST should be used in a security process. It finds vulnerabilities that are visible only when the app is running. It's language-independent and works across different platforms. Apps can be tested without reviewing the code. It helps with compliance by validating runtime behavior, and is ideal for both blackbox and greybox testing scenarios. DAST limitation While DAST is powerful, it's not perfect. Because it doesn't look at code, it may miss internal issues or logic-based bugs that a static analysis would catch. It also depends on having a running application, a working test, or a staging app, so it's not suitable for early development. You might also see false positives or incomplete test coverage if the tool cannot explore all app paths. That's why DAST is often combined with other testing methods. Here are some of the most widely used DAST tools. OWASP Zap is a free open source and is beginner-friendly. WebSuite is a powerful toolkit, often used by penetration testers. Acunetix and NetSparkers are great for enterprise use. IBM AppScan and Coalesce Web Application Scanner are suitable for large-scale security teams. These tools can automate many of the steps involved in scanning and identifying vulnerability in real-world applications. To summarize, DAST checks apps while they run and mimics real attacks. Black-box testing is realistic while limited in scope. It reflects the external attacker's view. Gray-box testing offers deeper insights with more context and has partial insight into the system. DAST is best used with other methods to secure apps truly and is effective for runtime flows. Ask yourself, in what situation would gray-box testing give better results than black-box testing?

The lesson details advanced graphical programming techniques using a coordinate system model. Initially, basic translation is demonstrated, where objects are moved along the X or Y axes. Advanced manipulation requires understanding the transformation of the coordinate system itself. To rotate objects, the coordinate system must be rotated by a specified angle around the origin.

A critical sequence involves applying both rotation and translation simultaneously. The order of these transformations is crucial; applying a translation first and then a rotation will yield a different result than rotating first and then translating. This sequential nature mandates careful programming of the mathematical order of operations.

Furthermore, the concepts extend to applying multiple, chained transformations sequentially. The system allows for complex graphical paths by combining translation, rotation, and scaling in a programmed order. For instance, one can rotate an object, scale it up, and then translate it to a new screen position.

The discussion emphasizes the inherent dependency between these transformation types. When multiple operations are chained, the final position of the object depends on how the system updates its internal coordinate state.

To manage complex drawing states, the programming environment provides stack-based methods. Saving the current transformation state allows the program to temporarily change the system's coordinates without affecting later drawings. Restoring the saved state reverts the entire system back to the exact coordinates it was in previously.

This ability to save and restore transformations makes it possible to draw complex, non-linear graphics composed of multiple independent components. The use of a stack is essential for maintaining program flow integrity during drawing operations.

Finally, the technique concludes by demonstrating how these cumulative transformations can define virtual, rotating drawing surfaces for interactive graphics, effectively allowing the user to manipulate the view space itself.

### 12_Object_Oriented_Programming_in_Javascript_(OOP)_Video_._Duration_13_minutes_13_min

# Object Oriented Programming in Javascript (OOP)
Video•
. Duration: 13 minutes
13 min - Summary

URL: https://www.coursera.org/learn/uol-graphics-programming/lecture/SEvXf/object-oriented-programming-in-javascript-oop

Object-Oriented Programming (OOP) is a paradigm built upon the concept of objects, which combine data and code. These objects structure data via fields (attributes/properties) and encapsulate behavior through methods (functions). Conceptually, a class acts like a "cookie cutter," defining the blueprint for creating multiple instances, or objects. An object represents an instance of that class, sharing structure but possessing unique attributes. Initially, a simple button example can be created using a traditional, non-OOP approach. This requires manually defining global variables like `state`, `locX`, `locY`, `width`, and `height`. Interactivity is implemented by defining methods that check boundaries, such as comparing mouse coordinates to the object's stored dimensions.

The primary weakness of this structure is scope management, as every piece of logic pollutes the global scope. The solution is object-oriented encapsulation. We define a `Button` class to encapsulate all related data and behavior. The `Button` constructor accepts necessary parameters (like dimensions and position) to initialize its private state. Key methods, like `draw()` and `isMouseOver()`, operate solely on the object's internal data.

The practical implementation involves instantiating the object: `let myButton = new Button(x, y, w, h);`. This keeps the code clean and modular. Furthermore, the class structure allows for complex interactions, such as handling clicks, which can involve checking if the mouse position falls within the button's defined area.

By adopting the class structure, we solve the problem of scope pollution, as the methods and properties only interact with the object instance they belong to. The system becomes highly scalable. We can create multiple, independent button objects—each retaining its own state—without interference. This model is much cleaner, reusable, and maintains strict boundaries between different components.

In summary, OOP structures encapsulate state and behavior into reusable modules, transforming scattered, global variables into organized, managed objects. This transition from procedural code to object-oriented code is fundamental to building robust, maintainable, and complex applications.

4. The `Ball` class constructor initializes essential properties: `speedX`, `speedY`, `locX`, and `locY`, often assigned random initial values.
5. Variables like `locX` and `locY` (location) and `speedX` and `speedY` (speed) track the ball's position and movement parameters.
6. Movement is handled by a `move` function, where the new location is calculated as the old location plus the speed, utilizing syntactic sugar (e.g., `this.locX += this.speedX`).
7. The program's logic is organized within a `run` function, which calls `draw`, `move`, and any necessary update/interaction functions sequentially for structured execution.
8. To implement bouncing, the code checks if `locX` is less than zero or greater than `width`, or if `locY` is outside the bounds of the canvas.

### 06_Introducing_collision_detection_Video_._Duration_7_minutes_7_min

# Introducing collision detection
Video•
. Duration: 7 minutes
7 min - Summary

URL: https://www.coursera.org/learn/uol-graphics-programming/lecture/IhuHQ/introducing-collision-detection

Collision detection is formally defined as the computational problem of determining the intersection of two or more objects. While often associated with video games, this technology also has critical applications in robotics and complex simulations. When simulating many rigid bodies, the collision detection problem quickly becomes computationally expensive. A naive approach requires comparing every shape against every other shape. For instance, with 10 shapes, this necessitates 100 comparisons, leading to high complexity. To manage this computational load, the process is strategically divided into two distinct phases: the broad phase and the narrow phase. The primary goal of the broad phase is to efficiently identify pairs of rigid bodies that *might* be colliding. This phase aims to exclude pairs that are certainly not overlapping. To optimize this process, physics engines utilize spatial partitioning techniques alongside bounding boxes. A bounding box simplifies comparison by establishing a simple check for potential overlap before expensive detailed analysis. Instead of testing all vertices, one first checks for overlap using the bounding boxes themselves. To minimize redundant bounding box tests, various space partitioning algorithms are employed. These structures include uniform grids, octrees for 3D space, spatial hashing, and sort and sweep. The quadtree is a specialized data structure used for dividing a 2D region into manageable subnodes. A quadtree starts as a single node and progressively splits into four subnodes as more objects are added. Objects are then placed into the appropriate subnode based on their coordinates within the 2D space. This structure allows the elimination of expensive collision checks between objects located in separate quadrants. For example, objects in the top-left node cannot collide with objects in the bottom-right node, saving significant computational effort. After efficient space partitioning, the process moves to the narrow phase. The narrow phase involves applying specific, detailed algorithms for the potential colliding pairs identified previously. Game physics engines employ unique detection algorithms for every pair of shape types, such as circles, edges, and convex polygons. For instance, checking if a triangle collides with a rectangle might involve iterating through all of the triangle’s vertices. A simple yet fundamental test involves determining if a point is contained within a circle. This condition is met if the distance between the circle’s center and the point is less than the circle’s radius. Extending this logic, two circles collide if the distance between their respective centers is less than the sum of their individual radii. Specifically, if $\text{Distance}(\text{Center}_1, \text{Center}_2) < \text{Radius}_1 + \text{Radius}_2$, a collision is confirmed. The overall sequence—broad phase $\rightarrow$ bounding boxes $\rightarrow$ space partitioning $\rightarrow$ narrow phase—greatly reduces the necessary number of computational comparisons.

### 06_Adding_and_deleting_multiple_bodies_Video_._Duration_13_minutes_13_min

# Adding and deleting multiple bodies
Video•
. Duration: 13 minutes
13 min - Summary

URL: https://www.coursera.org/learn/uol-graphics-programming/lecture/LCeRH/adding-and-deleting-multiple-bodies

This process details how to manage the lifecycle of dynamically generated physical objects (like falling blocks or projectiles) within a simulation environment, focusing on efficient object cleanup to prevent performance degradation.

The core concept involves three stages: **Initialization**, **Simulation/Rendering Loop Management**, and **Cleanup/Deactivation**.

### 1. Object Initialization and Structure

When a new object is created, it must be initialized in the physics engine's world.

**A. Object Definition:**
The object is instantiated, receiving properties like position, velocity, and dimensions.

**B. Integration into the Simulation World:**
Crucially, the object must be added to the physics world:
$$\text{world.add}(\text{box})$$
This step allows the engine to track the object's physical state (applying gravity, calculating collisions, etc.) during the simulation update cycle.

**C. Storing References:**
The object reference must be stored in a dedicated array or list (e.g., `activeObjects`) to ensure it is accessible for the main game loop iteration.

### 2. Simulation Loop Management (The Tick)

In the main loop, physics updates and drawing occur. The most critical part of this phase is determining if an object has finished its useful life.

**A. Physics Update:**
The simulation world must be advanced for a fixed time step ($\Delta t$):
$$\text{world.step}(\Delta t)$$
This calculation updates the position and velocity of all active objects.

**B. Detection of Termination Condition:**
After the physics step, every active object must be checked against its termination criteria. Common conditions include:
1. **Collision:** The object hits a designated "ground" or "safe" object.
2. **Boundary:** The object falls outside the visible or simulation-defined boundaries (e.g., below $Y=-100$).
3. **Time Limit:** The object has existed for too long.

**C. Iteration and Flagging:**
A loop iterates over the `activeObjects` list. Any object that satisfies the termination condition is flagged for removal.

### 3. Cleanup and Deactivation (The Key Optimization)

If an object is merely flagged, it remains in the physics world, consuming computational resources even if it is invisible. Proper cleanup requires removing the object from the **simulation world** *and* the **active object list**.

**A. Removing from the World (Crucial Step):**
The flagged object must be explicitly removed from the physics world:
$$\text{world.remove}(\text{box})$$
This step immediately tells the physics engine to stop calculating forces, collisions, and movements for that specific object, drastically improving performance.

**B. Deleting the Reference:**
The object reference must then be removed from the application's list of active objects (`activeObjects.remove(box)`). This prevents the object from being processed in future iterations of the loop.

### Summary of the Optimized Workflow

The overall process within the game loop must look like this:

$$\text{For each box in activeObjects:}$$
$$\quad \text{world.step}(\Delta t) \text{ (Update all boxes)}$$
$$\quad \text{If (box hits ground) OR (box out of bounds):}$$
$$\quad \quad \text{world.remove}(\text{box}) \text{ (Stop simulation calculations)}$$
$$\quad \quad \text{activeObjects.remove}(\text{box}) \text{ (Remove from iteration list)}$$

By systematically removing the object from the physics world upon its demise, we prevent "zombie objects"—entities that are no longer visible or relevant but continue to consume CPU cycles in background physics calculations.

Randomness is established as a fundamental pillar in generative processes, utilized in domains like art and game design. A truly random process, by definition, generates unpredictable and unbiased outcomes, such as the expected uniform distribution when throwing a fair die. The apparent randomness of physical objects, like dice, actually stems from their extreme sensitivity to tiny variations in initial conditions. To achieve true randomness, one must measure physical phenomena guaranteed to be unpredictable. Example sources include atmospheric noise, thermal noise, and measuring radioactive material decay. In contrast, computers cannot generate true randomness; instead, they produce pseudorandom numbers (PRNs). PRNs are numbers that merely *look* random but are actually completely deterministic. They are generated by an algorithm that relies on a shorter initial value, known as the seed or key. Consequently, knowing the seed allows for the perfect reproduction of the entire supposedly random sequence. The pseudorandom number generator often uses system information, such as the current time, to determine its initial seed. In programming, the function `random()` accepts optional parameters, typically defining minimum and maximum values, such as `random(0, width)`. To plot points, coordinates are often generated using independent calls like `randX` and `randY`. The code demonstrates that simply using random calls for size, color, and position results in independent variables. If variables are needed to maintain a uniform shape (like a circle), the value must be stored in a local variable, such as `var size = random(...)`, and reused. Crucially, the `randomSeed()` function allows the user to manually set the starting point for the random function. Setting a specific seed, such as `randomSeed(50)`, ensures that the exact same pattern of "random" values will emerge every time the program is reloaded. This deterministic behavior is useful when generating consistent generative art pieces. Randomness is applied widely, deciding object position, size, color, or applying random offsets to carefully selected values. It can also be used to simulate movement, such as in a random walk, or to determine when a random event occurs. For example, probability can be calculated using range checks on a random variable $R$, where $P(X)$ occurs if $R < 0.3$. Although useful, random coordinates are not always appropriate for modeling nature. # Introduction to materials and lights

The following summary encapsulates the core concepts of 3D graphics lighting, material properties, and the functionality of modern rendering engines, structured to maintain technical accuracy and flow.

***

### I. Foundations of Rendering and Materials

Rendering engines are responsible for calculating how light interacts with objects to produce a final image. The fundamental difference between traditional 2D graphics and 3D rendering is the complex mathematical simulation of light sources and object surfaces.

*   **Material Properties:** The *material* dictates how an object reacts to light. Properties like diffuse color, specular highlights (shininess), and reflectivity are key parameters.
*   **The Engine’s Role:** The rendering engine interprets these material properties and the light sources to simulate global illumination, allowing for realistic shading and object appearance without manual calculation for every point on a surface.

### II. Types of Light Sources

Light sources are categorized by how they emit illumination, which affects how shadows and highlights are cast:

1.  **Ambient Light:** A soft, general light that fills all areas of a scene evenly. It provides basic visibility to prevent areas from appearing completely black and is independent of location or time.
2.  **Directional Light:** Simulates distant sources, such as the sun. Since the light rays are parallel, this light type is excellent for creating consistent, uniform shadows across a wide area, regardless of the object's position in the scene.
3.  **Point Light:** Emits light equally in all directions from a single, fixed location (like a bare bulb). Its intensity decreases with the square of the distance from the source, creating realistic falloff and volume.
4.  **Spot Light:** A specialized point light that emits light within a defined cone. This allows for highly controlled, focused illumination, ideal for stage lighting or spotlights, and typically includes parameters for beam angle and falloff.

### III. Advanced Light Interaction (The Control of Color and Intensity)

Understanding how to control light requires manipulating color, intensity, and reach:

*   **Color Temperature:** Light sources can be assigned specific colors (e.g., warm yellow for incandescent bulbs, cool blue for daylight) to affect the mood and perceived time of the scene.
*   **Intensity and Range:** Light sources have configurable intensity values and, crucially, a *range* or *attenuation* setting. This dictates how far the light travels before its effect drops off to zero, which is essential for creating believable localized lighting.
*   **Color Blending:** When multiple light sources overlap, the engine must blend their respective colors. For instance, a red spotlight overlapping with a cool blue directional light will result in a resultant purple hue, demonstrating the additive nature of light.

### IV. Dynamic Environment Control

To achieve a cinematic look, developers use advanced techniques to manage the overall lighting environment:

*   **Global Illumination (GI):** This is a sophisticated calculation that simulates light bouncing off surfaces (indirect lighting). Instead of light only coming directly from a source, GI accounts for secondary and tertiary bounces, making shadows softer and allowing colors to subtly reflect onto adjacent, non-illuminated objects (e.g., a red wall casting a faint red glow on the adjacent white floor).
*   **Exposure Control:** The virtual camera’s *exposure* setting controls the overall brightness of the resulting image, compensating for scenes that might be artificially too dark or too bright after all light calculations have been performed.

A bitmap, or raster image, is defined as a rectangular array of pixels and is inherently resolution-dependent, meaning zooming in indefinitely is not possible. Bitmaps are also categorized by their compression methods: lossless or lossy. Lossless compression, exemplified by PNG, uses algorithms to represent data in a way that reduces file size without losing information. For example, instead of storing the full RGB values, lossless compression might store the difference between consecutive pixels' channel values, which are typically smaller numbers and require fewer bits. This allows the original image data to be reconstructed without any loss of fidelity.

A standard HD image, such as 1920 by 1080 pixels (only two megapixels), contains about 2 million pixels, which requires multiplying by four channels (RGB and Alpha) to reach approximately 8 megabytes of data. PNG was developed as an open alternative to GIF, supporting lossless compression and crucial transparency via the alpha channel. Instead of just three bytes for color, four bytes are used to describe the alpha channel, which dictates the amount of transparency.

The JPEG format represents a lossy compression method, commonly used by professional photographers and digital cameras. JPEG works by leveraging how the human brain perceives color, allowing certain information to be discarded without noticeable degradation. Vector images, conversely, are not based on pixel arrays but use mathematical formulas to describe geometric objects like circles and polygons. When describing shapes, a vector file contains explicit coordinates (x, y), dimensions (width, height), and stroke details, similar to p5js object descriptions.

Calculating the index for a pixel involves mapping 2D coordinates $(x, y)$ to a 1D array index, analogous to calculating a date based on a known month width. Accessing pixel values one by one using functions like `img.get(x, y)` is computationally slow, particularly when processing large numbers of pixels. If the red channel of pixel 0 is at index 0, the green channel of pixel 0 is at index 1, the blue is at index 2, and the alpha is at index 3. For the second pixel (starting at index 4), the green channel is located at index 5. Therefore, retrieving the color values for a specific pixel $(x, y)$ requires calculating the starting index and then applying the appropriate offset for the desired channel. Utilizing these direct array accesses, developers can efficiently read $R = \text{data}[i], G = \text{data}[i+1], B = \text{data}[i+2]$, etc., which is vital for efficient image manipulation.

## Technical Deep Dive: The Mathematics of Blurring

### What is Convolution?

In simple terms, convolution measures the **local relationship** between the input image and the kernel.

Given an input image $I(x, y)$ and a kernel $K(i, j)$, the output pixel value $O(x, y)$ is calculated by:

$$
O(x, y) = \sum_{i=-k/2}^{k/2} \sum_{j=-k/2}^{k/2} I(x-i, y-j) \cdot K(i, j)
$$

Where:
* $k$ is the size of the kernel (e.g., $k=3$).
* $(x, y)$ is the coordinate of the output pixel.
* $K(i, j)$ are the weights of the kernel.

### Example: Gaussian Blur Kernel
While the speaker uses a simple average kernel, professional blurring often uses a **Gaussian Kernel**. This kernel assigns higher weights to the center pixel and lower weights to the outer pixels, creating a more natural-looking blur.

Instead of filling the kernel with equal values (like $1/9$), the values in a Gaussian kernel follow a bell curve distribution, which mimics how light scatters naturally.

**4. Practical Challenges and Refinements:**

**Background Subtraction: A Comprehensive Overview**

Background Subtraction is a foundational technique in computer vision used to isolate moving objects (foreground) from a static or slowly changing scene (background) within a video stream. The core principle relies on modeling what constitutes "normal" or quiescent data—the background—and flagging any significant deviation from this model as the object of interest.

**Technical Methodology and Implementation**

The process operates by establishing a baseline model of the background environment, pixel by pixel. When new frames arrive, the system compares the current frame's pixel values against the stored background model. This comparison is not absolute; rather, it involves a statistical comparison across multiple frames to account for minor variations due to natural noise or slight camera jitters.

If the difference between the current pixel value and the expected background value exceeds a predefined threshold, the pixel is classified as foreground. This mechanism generates a binary mask, where white pixels represent detected motion or objects, and black pixels represent the presumed background.

**Algorithmic Refinements and Stability**

To enhance robustness, advanced algorithms incorporate temporal filtering. Simple pixel-by-pixel subtraction can be highly susceptible to noise or sudden, non-object-related lighting changes. Therefore, sophisticated implementations employ methods like Gaussian Mixture Models (GMMs) which adaptively update the background model, allowing it to slowly accommodate gradual changes (e.g., changing seasons or slight camera drift) without erroneously flagging them as foreground.

**Code Structures and Data Flow**

In practical programming terms, the process involves three key components:

1.  **The Background Model:** A data structure (e.g., a statistical model) that stores the probability distribution of pixel intensities for every location in the scene over time.
2.  **The Difference Calculation:** A mathematical operation comparing the current frame ($\text{Frame}_t$) to the expected background ($\text{BackgroundModel}$).
3.  **Thresholding and Mask Generation:** Applying a threshold ($\tau$) to the resulting difference array ($\text{Difference} = |\text{Frame}_t - \text{BackgroundModel}|$). Any $\text{Difference} > \tau$ triggers a foreground event, populating the mask.

**Challenges and Limitations**

While powerful, Background Subtraction faces inherent challenges. The primary limitation is the assumption that the background is entirely static or predictable. If the camera or lighting conditions undergo abrupt, radical changes (e.g., moving from indoors to outdoors), the existing background model becomes invalid, leading to significant false-positive detections. Furthermore, shadowing or partial occlusions can confuse the system, requiring manual tuning of the $\tau$ value for optimal performance.

In summary, Background Subtraction provides a highly effective framework for motion detection, transforming raw video data into meaningful, segmented data streams that enable subsequent tasks such as object tracking, counting, and behavioral analysis.

# Optical flow
Video•
. Duration: 11 minutes
11 min - Summary

URL: https://www.coursera.org/learn/uol-graphics-programming/lecture/6w1Bg/optical-flow

1. Optical flow represents the apparent motion of pixels within an image, often serving as an accurate approximation of true physical movement.
2. Unlike techniques that process individual pixels or analyze single-frame pixel differences, optical flow employs medium-level techniques by processing entire batches of pixels together.
3. It provides a detailed description of moving regions and quantifies the velocity of that motion, visualized through movement vectors.
4. The underlying assumption of optical flow algorithms is that the color or intensity of a specific pixel remains invariant from one video frame to the next.
5. This means the algorithm fundamentally assumes that a pixel, while moving, retains its original brightness and intensity across sequential frames.
6. However, calculating optical flow is susceptible to noise and sudden illumination changes, such as when sunlight rapidly shifts behind clouds.
7. Furthermore, a key technical limitation is the assumption that the movement between frames must be relatively small.
8. Although cameras recording at 25 frames per second usually capture minor movements, very fast-moving objects can cause the algorithm to fail in finding accurate pixel matches.
9. Despite these limitations, optical flow has broad applications, making it critical for automated traffic surveillance to detect and track vehicles.
10. Beyond tracking stationary objects, it can calculate global motion if the camera itself is moving, allowing for stability analysis.
11. Drone footage, for instance, uses optical flow to understand global scene movement and stabilize the resulting imagery, counteracting shake.
12. Conceptually, using optical flow information allows for an estimation of depth in 3D space, as objects closer to the camera appear to move faster.
13. This depth understanding is crucial for devices like smart vacuum cleaners, enabling them to map environments using only a single 2D camera.
14. Another application is using optical flow as a precursor to image segmentation, for example, identifying moving objects in front of a static background.
15. In video compression, optical flow exploits temporal redundancy by encoding movement vectors between frames rather than describing every pixel individually.
16. This principle is central to standards like MPEG, which utilizes B and P frames to encode movement information for maximum compression.
17. Technically, the algorithm searches for corresponding pixels in the neighborhood of the current pixel across two consecutive frames.
18. Processing two frames yields two distinct vector fields: $\mathbf{U}$ (for movement on the x-axis) and $\mathbf{V}$ (for movement on the y-axis).
19. The implementation provided relies on an external library (the `flow.js` library) rather than being coded from scratch.
20. The workflow involves loading the current frame and checking for existing data from the previous frame to calculate the flow.
21. If the frames are identical, the system correctly ignores the calculation to prevent unnecessary processing.
22. The output provides the vectors $\mathbf{U}$ and $\mathbf{V}$, which are then looped over and processed to display the calculated movement.
23. To reduce visual noise, a threshold is applied to the absolute length of the movement vector, filtering out minor or random pixel jitters.
24. Additionally, the system calculates an average movement vector, which can be scaled and utilized to control graphical elements or predict movement direction.
25. Ultimately, optical flow provides a powerful, foundational understanding of motion, which can then be used for advanced control, visualization, and data recovery across multiple domains.

This advanced computer vision segment focuses on moving beyond individual pixel or neighborhood techniques to process much larger patches using complex algorithms. The face is a particularly challenging object for computer vision, as humans possess a specialized circuitry for its recognition, suggesting its evolutionary and cognitive importance. Evidence for this specialization is seen in conditions like prosopagnosia, where individuals lose the ability to make global face identification despite normal vision. Our brains must recognize faces despite viewing changes, such as varying illumination or extreme viewing angles. Furthermore, faces are elastic, constantly changing shape due to expressions, talking, or eating. The algorithm must also overcome temporal changes, accounting for weight loss, aging, or changing accessories.

Computer-based detection algorithms must replicate this robustness to be commercially reliable, a field that has progressed dramatically (e.g., Google and Facebook's facial tagging). It is crucial to distinguish between face detection and face recognition: detection simply locates the face in an image, while recognition identifies the specific individual. The process presented utilizes the Viola-Jones algorithm, first introduced in 2001, which revolutionized the field by enabling real-time object detection. This algorithm exploits the fact that all human faces share measurable, repeatable properties, managed through hard features.

The development process begins by collecting extensive sets of positive images (containing faces) and negative images (not containing faces). Features are extracted by using these properties to match characteristics, often by examining adjacent rectangular regions within a specific area. The core mechanism involves calculating pixel intensities across these regions and determining the difference between the summed values. A key example is the Haar feature, which relies on the measurable property that the eye region is typically darker than the bridge of the nose. This process of feature extraction can gather hundreds of thousands of features, requiring training to select the most effective features for the classifier.

The practical implementation involves loading an external library, such as `objectdetect`, which provides an implementation of the Viola-Jones framework. This approach requires initializing a classifier, specifying the desired object (e.g., "frontal face"), and defining input dimensions (width, height) and a scale factor. The system then copies camera information into a dedicated image and feeds it to the face detector. The detector outputs results into an array, containing coordinates and a crucial confidence score for each identified face. The code then iterates over this array, creating a rectangle only when the confidence score exceeds a specified threshold (e.g., 4).

The initial implementation speed can be slow due to the algorithm's age and specific overhead, necessitating scaling down the input image size to improve performance. Although the model is trained for "frontal faces" only, the system demonstrates remarkable ability to accurately locate the user's face in real-time. The final output provides precise bounding box information, which can then be applied to other, larger images or subsequent processing steps. This entire framework shows how machine learning techniques are employed to solve complex visual recognition problems.
